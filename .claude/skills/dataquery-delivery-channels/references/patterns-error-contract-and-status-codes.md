> DataQuery/Fusion re-engineering (Project 3) delivery layer. Stack: **Python 3.12 · FastAPI · Pydantic

## 0. The one-paragraph thesis (read this first)

A data API's error contract is part of its **public type system**, exactly like its success schema. If
`GET /series/FOO` returns `{"error": "boom"}` one day and an HTML stack trace the next, every consumer —
the Python SDK, the agent tool loop, a notebook, a partner integration — has to special-case both, and
the generated SDK cannot turn either into a typed exception. The fix is a **single error envelope, a
single status-code map, and a single set of machine codes**, defined once, declared in OpenAPI, and
emitted by *every* channel. On top of that, this product line adds two domain-specific terminal states
that are **NOT generic HTTP errors** — `needsKey` (the upstream provider credential is missing) and
`unavailable` (the upstream/store could not serve this number right now). They are first-class because
the alternative — inventing or backfilling a number to "look complete" — is the single worst failure a
financial-data product can ship. We model them as *data*, not as `500`s.

We anchor the envelope on **RFC 9457 Problem Details** (the IETF standard, the wire shape), borrow
Stripe's **machine `code` + `type` + `param`** discipline (the most-copied error taxonomy in the
industry), borrow JSON:API's **array-of-errors** for multi-series partial failure, and implement it with
**FastAPI exception handlers** so the boundary is one place, not scattered `raise`s.

## 1. RFC 9457 — the wire shape we build on

RFC 9457 *Problem Details for HTTP APIs* (IETF, Aug 2023; obsoletes RFC 7807) is the standard for
machine-readable HTTP error bodies. Its purpose, verbatim: to "define a 'problem detail' to carry
machine-readable details of errors in HTTP response content to avoid the need to define new error
response formats for HTTP APIs." Source: <https://www.rfc-editor.org/rfc/rfc9457.html>.

### 1.1 The members (normative)

From RFC 9457 §3.1, the standard members of the problem-details object, each quoted from the spec:

| Member | Type | Definition (verbatim, RFC 9457 §3.1) |
|---|---|---|
| `type` | string (URI) | "a URI reference … that identifies the problem type. Consumers MUST use the 'type' URI (after resolution, if necessary) as the problem type's primary identifier." Absent ⇒ "its value is assumed to be `about:blank`." |
| `status` | number | "the HTTP status code … generated by the origin server for this occurrence of the problem." And: "Generators MUST use the same status code in the actual HTTP response." |
| `title` | string | "a short, human-readable summary of the problem type. It SHOULD NOT change from occurrence to occurrence of the problem, except for localization." |
| `detail` | string | "a human-readable explanation specific to this occurrence of the problem." Should "focus on helping the client correct the problem, rather than giving debugging information." |
| `instance` | string (URI) | "a URI reference that identifies the specific occurrence of the problem." |

Two rules that bite us directly:

1. **`type` is the primary key, not `status`.** RFC 9457 §3 is explicit that a consumer "MUST use the
   'type' URI … as the problem type's primary identifier" and "MUST NOT" infer more from the status code
   alone — many problem types can share one status (a `400` can be `unknown-frequency`, `range-inverted`,
   or `too-many-expressions`). This is why we add a stable **machine `code`** (§3) and key the SDK's
   exception classes off it, not off the bare `400`.
2. **Extension members are first-class.** RFC 9457 §3.2: "Problem type definitions MAY extend the problem
   details object with additional members" and "Clients consuming problem details MUST ignore any such
   extensions that they don't recognize." This is the hook that lets us carry `code`, `retryable`,
   `requestId`, `details`, and the per-series array **inside the standard envelope** without violating it.

Media type: `application/problem+json` when serialized as JSON (RFC 9457 §3). A consumer that does
content negotiation can branch on this media type to know "this is a structured error, parse it."

### 1.2 Why we don't ship *raw* RFC 9457 and stop there

RFC 9457 deliberately leaves `code`, `retryable`, and partial-success **undefined** — it is a container,
not a taxonomy. The widely-cited gap (see the Speakeasy and OpenAPI-Generator threads in §7) is that an
SDK generator can map `application/problem+json` to a generic exception, but with no stable machine code
it cannot give the consumer a *typed* `LicenseDeniedError` vs `UnknownSeriesError`. So our envelope is
**RFC 9457 + Stripe-style `code`/`type` extensions + JSON:API-style array** — standard on the wire,
ergonomic in the SDK.

## 2. The borrowed taxonomies (what each contributes)

### 2.1 Stripe — the machine `code` + structured fields

Stripe's error object (the most-copied API-error design in the industry) is:
`type`, `code`, `message`, `param`, plus context fields. Source: <https://docs.stripe.com/api/errors>.

Fields we adopt the *shape* of (Stripe's exact descriptions):

- `type` (enum): "The type of error returned. One of `api_error`, `card_error`, `idempotency_error`, or
  `invalid_request_error`." — we keep `type` as a **coarse category** (our values differ, §3.2).
- `code` (string, nullable): "For some errors that could be handled programmatically, a short string
  indicating the error code reported." — this is the load-bearing field; we make it **required**, not
  nullable.
- `message` (string, nullable): "A human-readable message providing more details about the error." — ours
  is required and safe to surface.
- `param` (string, nullable): "If the error is parameter-specific, the parameter related to the error." —
  we generalize to `details.field` / a JSON pointer.
- `doc_url`, `request_log_url` — we map to `details.docUrl` and `requestId`.

Stripe's `type → HTTP status` map (verbatim from the docs) is our template:

| Stripe type | HTTP status | Stripe's words |
|---|---|---|
| `invalid_request_error` | `400` | "arise when your request has invalid parameters." |
| `card_error` | `402` | (payment-specific; N/A here) |
| `idempotency_error` | `409` | "occur when an `Idempotency-Key` is re-used on a request that does not match the first request's API endpoint and parameters." |
| `api_error` | `500/502/503/504` | "any other type of problem (e.g., a temporary problem with Stripe's servers)… extremely uncommon." |

The `idempotency_error → 409` mapping is the precedent for our §6.

### 2.2 JSON:API — the array-of-errors for partial failure

JSON:API returns errors as an **array** under the top-level `errors` key; an error object "MAY have"
`id`, `status`, `code`, `title`, `detail`, `source` (`pointer`/`parameter`/`header`), `links`, `meta`,
and "MUST contain at least one of" them. Source: <https://jsonapi.org/format/#errors>. The rule we
borrow: "Servers may continue processing after encountering problems and return multiple error objects in
a single response." That is the exact mechanic our **multi-series partial-success** needs (§5): the batch
endpoint keeps going and reports a status *per expression*.

### 2.3 The synthesis

- **Single error** (whole request failed): RFC 9457 single object + our extensions (`code`, `retryable`,
  `requestId`, `details`).
- **Multi-series request** (some expressions succeeded, some failed): a `200`/`207` success envelope
  whose `series[]` each carry a per-series `status` (`ok` | `needsKey` | `unavailable` | `error`),
  JSON:API-style. We do NOT fail the whole batch because one ticker 404'd.

## 3. The envelope (the canonical contract)

### 3.1 The single-error JSON shape (on the wire)

```jsonc
// Content-Type: application/problem+json   (RFC 9457 §3)
// HTTP status line carries the SAME number as "status" below (RFC 9457: "MUST use the same status code")
{
  // ── RFC 9457 standard members ──────────────────────────────────────────
  "type":   "https://errors.dataquery.example/license-denied", // stable URI, the primary key
  "title":  "License denied",            // stable per type; never per-occurrence
  "status": 403,                          // mirrors the HTTP status line
  "detail": "Series 'YH:^TNX' is RED for redistribution; commercialOk=false.", // this occurrence
  "instance": "/series/YH:%5ETNX",        // what was requested

  // ── our extension members (RFC 9457 §3.2 — clients ignore unknown) ──────
  "error": {
    "code":      "license_denied",        // REQUIRED stable machine code — SDK keys exceptions off THIS
    "message":   "Series 'YH:^TNX' is not licensed for commercial display.",
    "retryable": false,                   // REQUIRED — is a blind retry of the SAME request useful?
    "requestId": "req_01J9Z8Q3M7B2",      // REQUIRED — correlate with server logs
    "details": {                          // OPTIONAL — code-specific, structured (never free-text blob)
      "series":    "YH:^TNX",
      "provider":  "yahoo",
      "commercialOk": false,
      "docUrl":    "https://docs.dataquery.example/errors/license_denied"
    }
  }
}
```

> Design note — why both the RFC 9457 top-level members **and** a nested `error` object? Two readers.
> RFC-9457-aware tooling (API gateways, generic problem-details client libs) reads `type`/`status`/
> `detail` at the top level. Our generated SDK and the agent tool layer read `error.code` /
> `error.retryable`. Carrying both costs ~120 bytes and satisfies both without a second format. The
> nested object also keeps our extensions namespaced so they never collide with a future RFC member.

### 3.2 The field contract (every field, why it exists, who reads it)

| Field | Required | Type | Stable? | Who reads it | Rule |
|---|---|---|---|---|---|
| `type` | yes | URI string | per problem-type | RFC-9457 tooling | One URI per `code`; resolvable to docs eventually. |
| `title` | yes | string | per problem-type | humans, logs | Same text for every occurrence of a code (RFC 9457 §3.1). |
| `status` | yes | int | — | gateways | MUST equal the HTTP status line (RFC 9457). |
| `detail` | yes | string | per occurrence | humans | This occurrence; **never** a stack trace, **never** a secret. |
| `instance` | no | URI string | per occurrence | logs | The path/resource requested. |
| `error.code` | **yes** | snake_case string | **stable forever** | **SDK, agent** | The contract. Renaming a code is a breaking change. |
| `error.message` | yes | string | per occurrence | humans | Safe to surface to an end-user. |
| `error.retryable` | **yes** | bool | — | **SDK retry loop** | `true` ⇒ the *same* request may succeed later (429/503). `false` ⇒ never (400/401/403/404). |
| `error.requestId` | **yes** | string | per occurrence | support, logs | One per request; in logs and the response. |
| `error.details` | no | object | — | SDK, humans | Structured, code-specific. A JSON pointer to the bad field, the unknown series id, the `Retry-After` echo, etc. |

**Stability law:** `error.code` values are a **public enum**. Add freely; never rename or repurpose. A
consumer's `except LicenseDeniedError` must keep meaning the same number forever. This is the same
discipline Stripe documents for its codes (programmatic handling depends on it).

### 3.3 The Pydantic v2 models (FastAPI source of truth)


### 3.4 A registry that ties `code → status → title → type → retryable`

A single table is the spine of the whole contract. Every handler looks the code up here; no handler
invents a status. This is the "one place" that makes the channels agree.

```python
# app/errors/registry.py
from dataclasses import dataclass
from .models import ErrorCode

BASE = "https://errors.dataquery.example"

@dataclass(frozen=True)
class Spec:
    status: int
    title: str
    retryable: bool
    @property
    def type_uri(self) -> str:  # type URI derived from the code → 1:1, documentable
        return f"{BASE}/{self.title.lower().replace(' ', '-')}"

REGISTRY: dict[ErrorCode, Spec] = {
    ErrorCode.bad_request:          Spec(400, "Bad request",            False),
    ErrorCode.unknown_frequency:    Spec(400, "Unknown frequency",      False),
    ErrorCode.range_inverted:       Spec(400, "Range inverted",         False),
    ErrorCode.too_many_expressions: Spec(400, "Too many expressions",   False),
    ErrorCode.invalid_expression:   Spec(400, "Invalid expression",     False),
    ErrorCode.auth_missing:         Spec(401, "Authentication required", False),
    ErrorCode.auth_expired:         Spec(401, "Authentication expired",  False),  # see §3.5: re-auth, not retry
    ErrorCode.license_denied:       Spec(403, "License denied",          False),
    ErrorCode.forbidden:            Spec(403, "Forbidden",               False),
    ErrorCode.unknown_series:       Spec(404, "Unknown series",          False),
    ErrorCode.unknown_dataset:      Spec(404, "Unknown dataset",         False),
    ErrorCode.conflict:             Spec(409, "Conflict",                False),
    ErrorCode.idempotency_conflict: Spec(409, "Idempotency conflict",    False),
    ErrorCode.validation_error:     Spec(422, "Validation error",        False),
    ErrorCode.rate_limited:         Spec(429, "Rate limited",            True),   # retry AFTER Retry-After
    ErrorCode.upstream_unavailable: Spec(503, "Upstream unavailable",    True),
    ErrorCode.store_unavailable:    Spec(503, "Store unavailable",       True),
}
```

### 3.5 `retryable` is per-code, and it is the SDK's whole retry policy

| Code (status) | `retryable` | Why |
|---|---|---|
| `bad_request`, `range_inverted`, … (400) | `false` | The request is malformed; re-sending it unchanged fails identically. |
| `auth_missing` (401) | `false` | No credential will appear by retrying; the *client* must add one. |
| `auth_expired` (401) | `false` | Retrying the **same** request fails; the client must **refresh the token first** then send a *new* request. `false` = "don't blind-retry"; it is not "give up". |
| `license_denied` (403) | `false` | A licensing verdict does not change on retry (commercial-ok gate, §4.2). |
| `unknown_series` (404) | `false` | The id doesn't exist; retrying won't create it. |
| `idempotency_conflict` (409) | `false` | Same key + different params is a *client bug*; retrying replays the error (Stripe behavior, §6). |
| `validation_error` (422) | `false` | Same as 400 — fix the body, don't retry it. |
| `rate_limited` (429) | `true` | Honor `Retry-After`, then the same request succeeds. |
| `upstream_unavailable`, `store_unavailable` (503) | `true` | Transient; backoff-and-retry is correct. |

> The distinction `retryable=false` ≠ "permanent failure". It means **"do not blind-retry the identical
> request."** `auth_expired` and `idempotency_conflict` are both `false` yet both have a corrective
> client action. The SDK encodes: `if not retryable: raise; else: backoff_and_retry()`. Anything
> requiring a *different* request (refresh token, new idempotency key) is a typed exception the caller
> catches, never an auto-retry.

## 4. The two domain-first-class states: `needsKey` and `unavailable`

This is the section that makes this product line's error contract different from a generic CRUD API's.
It exists because of **repo non-negotiable #1**: *never invent a finance number; a failed fetch returns a
typed `unavailable`/`needsKey`, never a fabricated value, never an HTML 500, never a RED-tier backfill to
"look complete."* (CLAUDE.md cross-cutting non-negotiable #1; `commercial-ok-gate.md` "In practice":
"Failed/over-budget fetches return typed `unavailable`/`needsKey` — never a fabricated value, never a
RED-tier backfill.")

### 4.1 Why these are NOT HTTP errors

The Lumina finance tools (`backend/finance/tools.ts`, read this session) already encode the principle the
right way. When the upstream provider key is missing the tool returns **data**, not a thrown error:

```ts
// backend/finance/tools.ts (Lumina, read 2026-06) — the pattern we port to Python
if (r.data.needsKey) {
  return { error: "Stock quotes are unavailable: TWELVE_DATA_API_KEY is not configured." };
}
// ...and on a rate-budget miss:
if (!res.ok) return { unavailable: "Live stock quotes are rate-limited right now — try again shortly." };
```

Notice: the tool's `execute()` **resolves** with `{ unavailable }` / `{ error }`. It does **not** throw,
does **not** fabricate a price, does **not** return a stale-looking zero. The model receives a typed
"no number available" and is grounded to say so. That is the behavior we are making a wire contract.

The reason these are not `500`s:

- A `500`/`503` says *"the server broke."* But "Twelve Data key is unset" or "this provider returned
  nothing for AAPL right now" is **not a server defect** — it is a **known, expected, named outcome** of a
  data pipeline that depends on third-party providers and credentials. Modeling a known outcome as an
  unexpected crash is a category error: it pages an on-call engineer for a configuration fact.
- The agent/SDK needs to **branch on it**: `needsKey` ⇒ "this requires a credential, tell the user / skip";
  `unavailable` ⇒ "no number now, say so or try a fallback provider". A bare `500` erases that distinction.
- Most importantly, on a **multi-series** request (§5) one series being `unavailable` must NOT fail the
  other nine. A `503` for the whole call would force the client to discard good data — pushing it toward
  the exact "backfill to look complete" anti-pattern the rule forbids.

### 4.2 Where each lives on the wire

| State | What it means | HTTP carrier | Why there |
|---|---|---|---|
| `needsKey` | The upstream provider credential for this fetch path is not configured. | **Per-series** field on a `200` batch (§5). For a single-series GET where the WHOLE dataset is keyless: a `200` with `{ "status": "needsKey", "series": [] }`. | It's a deterministic, expected data outcome; the request was well-formed and authorized. |
| `unavailable` | The number could not be fetched right now (provider down, over rate budget, store timeout, empty result). | Per-series field on a `200` batch. If the **entire** request can't be served and there is no partial data: **`503` with `error.code = upstream_unavailable`** (retryable). | A whole-request hard failure is a `503`; a per-series soft failure is data. |
| `license_denied` | The series exists and was fetched but `commercialOk=false` for redistribution. | **`403`** single-error (`error.code = license_denied`) for a redistribution endpoint; OR per-series `status:"license_denied"` on a batch. | Licensing is an authorization verdict, not a data outage. It maps to 403, not to `unavailable`. |

> The three are distinct and must never be collapsed. `needsKey` (we lack a credential) ≠ `unavailable`
> (we have the credential, the fetch failed) ≠ `license_denied` (we have the data, we may not redistribute
> it). Collapsing them hides the *fix*: add a key vs retry later vs purchase a license. The
> `commercial-ok-gate.md` rule is explicit that a number can be GREEN-but-wrong or RED-but-accessible —
> licensing is orthogonal to availability, so it gets its own state.

### 4.3 The per-series status enum (the data-plane terminal states)

```python
# app/series/models.py
from enum import Enum
from typing import Any
from pydantic import BaseModel

class SeriesStatus(str, Enum):
    ok            = "ok"             # points present, grounded, licensed
    needs_key     = "needsKey"       # provider credential missing — NOT an error
    unavailable   = "unavailable"    # fetch failed / over budget / empty — NOT an error
    license_denied = "license_denied" # commercialOk=false for redistribution
    not_found     = "not_found"      # series id does not exist in this batch

class Provenance(BaseModel):
    provider: str
    commercialOk: bool = False        # DEFAULT false (commercial-ok-gate.md)
    fetchedAt: str | None = None
    attribution: str | None = None    # rendered when a CC-BY / required-attribution source is GREEN

class SeriesResult(BaseModel):
    expression: str                   # the series/expression this row answers
    status: SeriesStatus
    points: list[dict[str, Any]] = [] # EMPTY unless status == ok. Never a fabricated value.
    provenance: Provenance | None = None
    message: str | None = None        # human note when status != ok ("TWELVE_DATA_API_KEY not set")
```

**The invariant `points == [] when status != ok` is the whole point.** There is no code path that emits a
non-`ok` status with a non-empty `points`. A fabricated, zero-filled, or backfilled point is impossible
to express in this model — which is exactly how the type system enforces non-negotiable #1.

## 5. Partial success on multi-series (the batch contract)

The DataQuery-class workload is *batch*: "give me `[AAPL.close, ^TNX, BTC-USD]` for the last year." One
of those can 404 or be rate-limited while the others are fine. JSON:API's rule applies: keep processing,
report per-item. Source: <https://jsonapi.org/format/#errors> ("Servers may continue processing after
encountering problems and return multiple error objects").

### 5.1 The envelope

```jsonc
// HTTP 200 (all ok) or 207 Multi-Status (mixed) — see §5.2 for the choice
{
  "requestId": "req_01J9Z8...",
  "series": [
    {
      "expression": "AAPL.close",
      "status": "ok",
      "points": [{"t":"2026-06-23","v":211.34}, /* … */],
      "provenance": {"provider":"twelvedata","commercialOk":false,"fetchedAt":"2026-06-24T13:00:00Z"}
    },
    {
      "expression": "^TNX",
      "status": "unavailable",          // soft failure — request keeps going
      "points": [],                      // EMPTY, never backfilled
      "message": "Upstream returned no data for ^TNX; try again shortly."
    },
    {
      "expression": "FRED:DGS10",
      "status": "needsKey",
      "points": [],
      "message": "FRED_API_KEY is not configured."
    }
  ],
  "summary": {"ok": 1, "needsKey": 1, "unavailable": 1, "license_denied": 0, "not_found": 0}
}
```

### 5.2 What status code does a *mixed* batch return?

There are three defensible choices; pick one and document it (this matters for the SDK):

| Choice | When the batch is "all ok" | When mixed | Pros / cons |
|---|---|---|---|
| **A. Always `200`** (recommended) | `200` | `200` + per-series status | Simplest for SDK (one happy path; inspect `series[].status`). The batch endpoint's *transport* succeeded; per-series outcomes are payload. **This is what Lumina's tools do.** |
| B. `200` / `207 Multi-Status` | `200` | `207` (RFC 4918) | Honest at the HTTP layer, but `207` confuses many HTTP clients and SDK generators; some retry middleware treats non-2xx-`200` oddly. Use only if a partner demands it. |
| C. `200` / `4xx` if ALL fail | `200` | `200` mixed; `4xx`/`503` only if **zero** succeeded | Hybrid: a batch where *every* series failed is a request-level failure (`503 upstream_unavailable` if all `unavailable`; `400` if all expressions invalid). Otherwise `200`. |

**Recommendation: A for the common case, with C's "all-failed ⇒ request-level error" as the one
exception** — if literally nothing could be served, returning a `200` with an all-empty body is a worse
experience than a `503 upstream_unavailable` that the SDK's retry loop can act on. Concretely:

```python
def batch_http_status(results: list[SeriesResult]) -> int:
    if all(r.status in {SeriesStatus.unavailable, SeriesStatus.not_found} for r in results) \
       and any(r.status == SeriesStatus.unavailable for r in results):
        return 503            # nothing servable AND at least one transient → retryable request-level
    return 200                # any ok, OR a deterministic all-needsKey/not_found → 200 payload
```

> The subtlety: an all-`needsKey` or all-`not_found` batch is **deterministic** (retrying won't help), so
> it should be a `200` payload the client reads once — NOT a `503` that triggers a pointless retry storm.
> Only an all-`unavailable` (transient) batch earns the retryable `503`. Encoding "is this worth
> retrying?" into the status is the same `retryable` discipline from §3.5, applied at the batch level.

## 6. Idempotency-Key handling (writes & job-create)

Reads (`GET`) are idempotent by definition. But this product line has **writes**: create-a-bulk-download
job, register-a-dataset, schedule-an-export. A client that times out and retries must not create two
jobs. We adopt Stripe's model verbatim where it fits. Source:
<https://docs.stripe.com/api/idempotent_requests>.

### 6.1 The rules we implement (Stripe, quoted)

1. **Header.** "Add `Idempotency-Key` to request options." We accept it on every non-`GET`/`DELETE`
   mutating endpoint.
2. **Result caching.** "Stripe saves the status code and body of the first request for any given
   idempotency key—whether it succeeds or fails. Subsequent requests with the same key return identical
   results, including `500` errors." We persist `(key) → (status, body, param_fingerprint, created_at)`.
3. **Scope.** "Only `POST` requests accept idempotency keys." We reject the header on `GET`/`DELETE` with
   a `400` (or ignore — pick one; we ignore, since they're already idempotent).
4. **Parameter validation = 409.** "The system compares incoming parameters to the original request. If
   parameters differ, an error is returned." Same key + **different** params ⇒
   `409 idempotency_conflict`. (Stripe: this is exactly its `idempotency_error → 409`.)
5. **Replay includes failures.** "A request that returns a `400` sends back the same `400` if followed by
   a new request with the same idempotency key." So our store caches errors too; we **replay** them.
6. **Retention.** "Keys are automatically pruned after 24 hours." We TTL the store at 24h.
7. **Key generation guidance** (for our SDK docs): "Use V4 UUIDs or random strings with sufficient
   entropy… Keep keys under 255 characters. Avoid sensitive data."
8. **Concurrency.** Stripe: results are saved only after execution begins; a concurrent in-flight request
   with the same key is safe to retry. We hold a short **lock** per key so two concurrent requests don't
   both execute (the second waits and replays the first's result).

### 6.2 The FastAPI dependency + store

```python
# app/idempotency.py
import hashlib, json, time
from typing import Any
from fastapi import Header, Request, HTTPException
from .errors.exceptions import ApiError
from .errors.models import ErrorCode

class IdemRecord:
    __slots__ = ("status", "body", "fingerprint", "created_at", "in_flight")
    def __init__(self, fingerprint: str):
        self.status = 0; self.body: Any = None
        self.fingerprint = fingerprint; self.created_at = time.time(); self.in_flight = True

# In production: Redis with a 24h TTL + SETNX lock. In-memory dict shown for clarity.
_STORE: dict[str, IdemRecord] = {}
TTL = 24 * 3600

def _fingerprint(method: str, path: str, body: bytes) -> str:
    # Stripe compares "API endpoint and parameters". Hash method+path+canonical-body.
    canon = json.dumps(json.loads(body or b"{}"), sort_keys=True, separators=(",", ":"))
    return hashlib.sha256(f"{method}\n{path}\n{canon}".encode()).hexdigest()

async def idempotency(
    request: Request,
    idempotency_key: str | None = Header(default=None, alias="Idempotency-Key"),
) -> "IdemGuard | None":
    if request.method in ("GET", "DELETE") or idempotency_key is None:
        return None  # idempotent by definition / not requested
    if len(idempotency_key) > 255:
        raise ApiError(ErrorCode.bad_request, "Idempotency-Key exceeds 255 chars.")
    body = await request.body()
    fp = _fingerprint(request.method, request.url.path, body)

    rec = _STORE.get(idempotency_key)
    if rec and (time.time() - rec.created_at) > TTL:
        rec = None; _STORE.pop(idempotency_key, None)         # pruned (>24h)
    if rec is not None:
        if rec.fingerprint != fp:                              # Stripe rule #4
            raise ApiError(
                ErrorCode.idempotency_conflict,
                "Idempotency-Key was reused with different parameters.",
                details={"key": idempotency_key},
            )
        if rec.in_flight:                                      # concurrent in-flight (Stripe #8)
            raise ApiError(
                ErrorCode.conflict,
                "A request with this Idempotency-Key is still in progress.",
                retryable=True,                                # safe to retry shortly
            )
        # Replay the saved status + body (Stripe #2/#5 — even if it was an error)
        raise _Replay(rec.status, rec.body)
    _STORE[idempotency_key] = IdemRecord(fp)
    return IdemGuard(idempotency_key)

class IdemGuard:
    """Handed to the route so it can save its result under the key when done."""
    def __init__(self, key: str): self.key = key
    def save(self, status: int, body: Any) -> None:
        rec = _STORE.get(self.key)
        if rec: rec.status, rec.body, rec.in_flight = status, body, False

class _Replay(Exception):
    def __init__(self, status: int, body: Any): self.status, self.body = status, body
```

Route usage and the replay handler:

```python
# in a write route
@app.post("/jobs", status_code=201)
async def create_job(spec: JobSpec, guard: IdemGuard | None = Depends(idempotency)):
    job = await jobs.create(spec)          # the real work, runs at most once per key
    body = job.model_dump()
    if guard: guard.save(201, body)         # cache status+body for replays
    return JSONResponse(status_code=201, content=body)

@app.exception_handler(_Replay)
async def replay_handler(request, exc: _Replay):
    # Return the EXACT first result, including its status (Stripe: "identical results, including 500s")
    return JSONResponse(status_code=exc.status, content=exc.body, headers={"Idempotent-Replayed": "true"})
```

> Production note: the in-memory `_STORE` is single-instance only. On the Fly multi-instance deploy this
> MUST be Redis (or the time-series store's Postgres) with `SET key val NX EX 86400` for the lock and a
> 24h TTL for the record — otherwise two instances each create a job. The lock (`in_flight` ⇒ `409
> retryable`) is what prevents the double-execute that a naive "check-then-write" race allows. This is the
> same atomic-guarded-write discipline R-SCALE §D requires for any contested write.

## 7. Documenting it in OpenAPI so the SDK gets typed exceptions

The whole point of a stable contract is that the **generated SDK** turns each `code` into a typed
exception (`LicenseDeniedError`, `RateLimitedError`, …) the caller can `except`. That only happens if the
OpenAPI document declares the error responses precisely. The known failure mode (OpenAPI-Generator issue
#20742, Kiota issue #4349) is that a vaguely-declared `application/problem+json` response yields one
generic exception with an unhelpful message. We avoid it by: one reusable schema, one response component
per status, the `code` enum in the schema, and a `code`-keyed mapping table the SDK template reads.

### 7.1 The OpenAPI fragment (FastAPI emits most of this from the Pydantic models)

### 7.2 How the SDK maps `code → exception`

The generator gives you a base `ApiError`. Layer a thin hand-written map keyed on `error.code` (the only
stable identifier — RFC 9457 §3 forbids inferring the type from `status` alone):

```python
# dataquery_sdk/errors.py (hand-written, sits over the generated client)
class DataQueryError(Exception):
    def __init__(self, code, message, *, retryable, request_id, status, details=None):
        super().__init__(message)
        self.code, self.retryable, self.request_id = code, retryable, request_id
        self.status, self.details = status, details or {}

class BadRequestError(DataQueryError): ...
class AuthError(DataQueryError): ...           # 401: auth_missing / auth_expired
class LicenseDeniedError(DataQueryError): ...  # 403
class NotFoundError(DataQueryError): ...       # 404
class IdempotencyConflictError(DataQueryError):...# 409
class ValidationError(DataQueryError): ...     # 422
class RateLimitedError(DataQueryError):        # 429
    def __init__(self, *a, retry_after: float | None = None, **k):
        super().__init__(*a, **k); self.retry_after = retry_after
class UnavailableError(DataQueryError): ...    # 503

_BY_CODE = {
    "bad_request": BadRequestError, "unknown_frequency": BadRequestError,
    "range_inverted": BadRequestError, "too_many_expressions": BadRequestError,
    "invalid_expression": BadRequestError,
    "auth_missing": AuthError, "auth_expired": AuthError,
    "license_denied": LicenseDeniedError, "forbidden": LicenseDeniedError,
    "unknown_series": NotFoundError, "unknown_dataset": NotFoundError,
    "conflict": IdempotencyConflictError, "idempotency_conflict": IdempotencyConflictError,
    "validation_error": ValidationError,
    "rate_limited": RateLimitedError,
    "upstream_unavailable": UnavailableError, "store_unavailable": UnavailableError,
}

def raise_for_problem(status: int, body: dict, headers: dict) -> None:
    err = body.get("error", {})
    code = err.get("code", "bad_request")
    cls = _BY_CODE.get(code, DataQueryError)
    kwargs = dict(message=err.get("message", ""), retryable=err.get("retryable", False),
                  request_id=err.get("requestId", ""), status=status, details=err.get("details"))
    if cls is RateLimitedError:
        ra = headers.get("Retry-After")
        kwargs["retry_after"] = float(ra) if ra and ra.isdigit() else None
    raise cls(code, **kwargs)
```

Caller ergonomics this buys:

```python
try:
    df = client.get_series("YH:^TNX")
except LicenseDeniedError:
    fall_back_to_green_source()          # licensing — a different fetch path
except RateLimitedError as e:
    time.sleep(e.retry_after or backoff())  # honor Retry-After
    df = client.get_series("YH:^TNX")
except AuthError:
    client.refresh_token(); ...           # re-auth then a NEW request
```

## 8. The status-code map (the full table, with the *why* for each)

This is the canonical mapping every channel obeys. Anchored on RFC 9110 status semantics and Stripe's map
(§2.1). RFC 9110 §15 is the normative source for the status meanings.

| Status | `error.code`(s) | When | `retryable` | Notes |
|---|---|---|---|---|
| **400** Bad Request | `bad_request`, `unknown_frequency`, `range_inverted`, `too_many_expressions`, `invalid_expression` | Param the server can reject by inspection: a `freq` not in the enum; `start > end`; more than the max expressions per batch; a malformed expression. | `false` | Distinct codes so the SDK/UI says *which* param. `unknown_frequency` carries `details.allowed`. |
| **401** Unauthorized | `auth_missing`, `auth_expired` | No `Authorization` header / token; or token expired. | `false` | `auth_expired` ⇒ client refreshes then sends a *new* request (§3.5). Include `WWW-Authenticate`. |
| **403** Forbidden | `license_denied`, `forbidden` | Authenticated but not permitted: `commercialOk=false` for a **redistribution** endpoint; or scope/entitlement missing. | `false` | License verdict is stable; never `unavailable`. `details.commercialOk:false`, `details.provider`. |
| **404** Not Found | `unknown_series`, `unknown_dataset` | The id/dataset does not exist. | `false` | A *known* series that's merely empty-right-now is **not** 404 — that's per-series `unavailable` (§4). |
| **409** Conflict | `conflict`, `idempotency_conflict` | Idempotency-Key reused with different params (Stripe → 409); optimistic-concurrency version mismatch; concurrent in-flight key. | `false` (in-flight: `true`) | §6. |
| **422** Unprocessable Entity | `validation_error` | Body is syntactically JSON but fails the schema (FastAPI `RequestValidationError`). | `false` | FastAPI's native 422; we reshape it into the envelope (§9.2). `details` = per-field list. |
| **429** Too Many Requests | `rate_limited` | The caller exceeded its rate budget (per-key/IP), or we shed load. | `true` | MUST send `Retry-After` (§8.1). |
| **503** Service Unavailable | `upstream_unavailable`, `store_unavailable` | A provider is down / circuit-open; the time-series store timed out; whole-request `unavailable` with no partial data. | `true` | `Retry-After` SHOULD be sent. Distinguishes "their fault" (`upstream_`) from "our store" (`store_`). |

**Deliberately NOT used as the primary failure path:** `500`. A `500` means an *unhandled* exception — a
bug, not a known outcome. Every *expected* failure above has a typed code and a non-500 status. A `500`
that escapes our handlers is itself a defect to fix, and the catch-all handler (§9.4) still wraps it in
the envelope (`code = "internal"`, `retryable=false`) so we never leak an HTML stack trace.

### 8.1 `Retry-After` on 429 / 503

`Retry-After` (RFC 9110 §10.2.3) takes **one of two forms** — an HTTP-date or a delay in **seconds**:

```
Retry-After: 3600              # delay-seconds form (preferred for APIs — unambiguous)
Retry-After: Wed, 24 Jun 2026 14:00:00 GMT   # HTTP-date form
```

MDN confirms 429 "may include" `Retry-After` "to indicate how long a client should wait before making the
request again." Source: <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/429>. We
always send the **delay-seconds** form (no clock-skew ambiguity), and **echo it into the body** so an SDK
that only parses JSON still sees it:

```python
def rate_limited_response(retry_after_s: int, request_id: str) -> JSONResponse:
    body = problem(ErrorCode.rate_limited,
                   detail=f"Rate limit exceeded; retry in {retry_after_s}s.",
                   request_id=request_id,
                   details={"retryAfterSeconds": retry_after_s})
    return JSONResponse(status_code=429, content=body,
                        media_type="application/problem+json",
                        headers={"Retry-After": str(retry_after_s)})
```

**Client side** (the SDK retry loop): honor `Retry-After` when present; otherwise fall back to
**exponential backoff with full jitter** to avoid the thundering-herd synchronization that identical
backoff causes. Sources: <https://zuplo.com/learning-center/http-429-too-many-requests-guide> and
<https://apistatuscheck.com/blog/api-rate-limiting-429-errors-production> ("respects the server's
Retry-After guidance when available, and falls back to jittered exponential backoff when it isn't";
"adding small randomness to delays (jitter) keeps everyone from retrying at once"). The JPM
`dataquery-sdk` itself ships `rate_limit_rpm` + `max_retries` + retry logic for exactly this
(<https://github.com/jpmorganchase/dataquery-sdk>).

```python
import random, time
def backoff_sleep(attempt: int, retry_after: float | None, *, base=0.5, cap=30.0):
    if retry_after is not None:
        time.sleep(retry_after); return            # server told us — obey it
    expo = min(cap, base * (2 ** attempt))
    time.sleep(random.uniform(0, expo))            # FULL jitter (uniform 0..expo), not fixed expo
```

## 9. Worked FastAPI exception handlers (the boundary, one place)

The contract is only deterministic if **one** layer produces it. We raise a single internal exception
`ApiError(code, message, …)` from anywhere in the service; one handler turns it into the envelope; and we
override FastAPI's two built-in handlers so *even the framework's* errors come out in our shape.

### 9.1 The internal exception + the problem builder

```python
# app/errors/exceptions.py
from typing import Any
from .models import ErrorCode
from .registry import REGISTRY

class ApiError(Exception):
    """Raise this anywhere. The handler maps it to the envelope via REGISTRY."""
    def __init__(self, code: ErrorCode, message: str, *,
                 detail: str | None = None, details: dict[str, Any] | None = None,
                 retryable: bool | None = None):
        spec = REGISTRY[code]
        self.code = code
        self.status = spec.status
        self.title = spec.title
        self.type_uri = spec.type_uri
        self.message = message
        self.detail = detail or message
        self.details = details
        self.retryable = spec.retryable if retryable is None else retryable
        super().__init__(message)
```

```python
# app/errors/render.py
from typing import Any
from .models import ErrorCode
from .registry import REGISTRY

def problem(code: ErrorCode, *, detail: str, request_id: str,
            details: dict[str, Any] | None = None, instance: str | None = None,
            message: str | None = None) -> dict[str, Any]:
    spec = REGISTRY[code]
    return {
        "type": spec.type_uri, "title": spec.title, "status": spec.status,
        "detail": detail, **({"instance": instance} if instance else {}),
        "error": {
            "code": code.value, "message": message or detail,
            "retryable": spec.retryable, "requestId": request_id,
            **({"details": details} if details else {}),
        },
    }
```

### 9.2 Register all four handlers in one place

```python
# app/errors/handlers.py
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from fastapi.exceptions import RequestValidationError
from starlette.exceptions import HTTPException as StarletteHTTPException
from .exceptions import ApiError
from .models import ErrorCode
from .render import problem

PROBLEM_JSON = "application/problem+json"

def _request_id(request: Request) -> str:
    # set by middleware (§9.3); always present
    return getattr(request.state, "request_id", "req_unknown")

def install_error_handlers(app: FastAPI) -> None:

    @app.exception_handler(ApiError)
    async def _api_error(request: Request, exc: ApiError):
        body = problem(exc.code, detail=exc.detail, request_id=_request_id(request),
                       details=exc.details, instance=str(request.url.path), message=exc.message)
        headers = {}
        if exc.code == ErrorCode.rate_limited and exc.details and "retryAfterSeconds" in exc.details:
            headers["Retry-After"] = str(exc.details["retryAfterSeconds"])
        if exc.code in (ErrorCode.auth_missing, ErrorCode.auth_expired):
            headers["WWW-Authenticate"] = "Bearer"
        return JSONResponse(status_code=exc.status, content=body,
                            media_type=PROBLEM_JSON, headers=headers)

    # FastAPI's native 422 → reshape into OUR envelope (don't leak the raw `detail` list shape)
    @app.exception_handler(RequestValidationError)
    async def _validation(request: Request, exc: RequestValidationError):
        # exc.errors() is the Pydantic list: [{"loc": [...], "msg": ..., "type": ...}, ...]
        fields = [{"path": list(e.get("loc", [])), "message": e.get("msg"), "type": e.get("type")}
                  for e in exc.errors()]
        body = problem(ErrorCode.validation_error,
                       detail="Request body failed validation.",
                       request_id=_request_id(request),
                       details={"fields": fields}, instance=str(request.url.path))
        return JSONResponse(status_code=422, content=body, media_type=PROBLEM_JSON)

    # Anything that raised a bare Starlette/FastAPI HTTPException (e.g. 404 from a missing route) →
    # wrap it so it's never a bare {"detail": "..."} that breaks the SDK's parser.
    @app.exception_handler(StarletteHTTPException)
    async def _http(request: Request, exc: StarletteHTTPException):
        code = {400: ErrorCode.bad_request, 401: ErrorCode.auth_missing,
                403: ErrorCode.forbidden, 404: ErrorCode.unknown_series,
                409: ErrorCode.conflict, 429: ErrorCode.rate_limited,
                503: ErrorCode.upstream_unavailable}.get(exc.status_code, ErrorCode.bad_request)
        body = problem(code, detail=str(exc.detail), request_id=_request_id(request),
                       instance=str(request.url.path))
        body["status"] = exc.status_code  # preserve the original status if it wasn't in the map
        return JSONResponse(status_code=exc.status_code, content=body, media_type=PROBLEM_JSON,
                            headers=getattr(exc, "headers", None) or {})

    # The catch-all: an UNHANDLED exception is a BUG. Wrap it, log the traceback, leak NOTHING.
    @app.exception_handler(Exception)
    async def _unhandled(request: Request, exc: Exception):
        rid = _request_id(request)
        # log.exception(...) here with rid; do NOT put the traceback in the body.
        body = {
            "type": "https://errors.dataquery.example/internal", "title": "Internal error",
            "status": 500, "detail": "An unexpected error occurred.",
            "instance": str(request.url.path),
            "error": {"code": "internal", "message": "An unexpected error occurred.",
                      "retryable": False, "requestId": rid},
        }
        return JSONResponse(status_code=500, content=body, media_type=PROBLEM_JSON)
```

The FastAPI handler mechanics here follow the official docs: custom handlers via `@app.exception_handler`,
overriding `RequestValidationError` and `StarletteHTTPException`, and returning `JSONResponse` with a
custom body/status. Source: <https://fastapi.tiangolo.com/tutorial/handling-errors/> (which shows the
default 422 `{"detail": [{"loc": ..., "msg": ..., "type": ...}]}` shape we reshape, and that you register
the **Starlette** `HTTPException` to catch both framework and FastAPI raises).

### 9.4 Raising it from a route / service (the domain states)

```python
# the place where needsKey / unavailable / license_denied are DECIDED
async def fetch_one(expr: str, prov: ProviderClient) -> SeriesResult:
    if not prov.has_credential:                  # needsKey — DATA, not an exception
        return SeriesResult(expression=expr, status=SeriesStatus.needs_key, points=[],
                            message=f"{prov.env_var} is not configured.")
    if not prov.commercial_ok and prov.endpoint_is_redistribution:
        # On a single-series redistribution endpoint, license is a 403:
        raise ApiError(ErrorCode.license_denied,
                       f"Series '{expr}' is not licensed for commercial display.",
                       details={"series": expr, "provider": prov.name, "commercialOk": False})
    try:
        points = await prov.fetch(expr)
    except UpstreamTimeout:
        return SeriesResult(expression=expr, status=SeriesStatus.unavailable, points=[],
                            message="Upstream timed out; try again shortly.")   # NOT a fabricated value
    if not points:
        return SeriesResult(expression=expr, status=SeriesStatus.unavailable, points=[],
                            message="No data returned for this series right now.")
    return SeriesResult(expression=expr, status=SeriesStatus.ok, points=points,
                        provenance=prov.provenance())
```

> The litmus test for "exception vs per-series data": **does the rest of the batch still make sense?** A
> malformed `freq` poisons the *whole* request → raise `ApiError` (400). One ticker being down → the
> other nine are fine → per-series `unavailable` (data). This split is precisely what lets us honor
> non-negotiable #1 *and* JSON:API partial success at the same time.

## 10. The gateway / streaming-channel equivalents

The same contract must hold when the response is **not** a single JSON body. Two cases.

### 10.1 The reverse-proxy / API-gateway equivalent

When a gateway (Nginx/Envoy/Cloudflare/Fly proxy) returns an error *before the app runs* (upstream
unreachable, request-size limit, gateway timeout), it emits its OWN body — typically HTML. That breaks
every SDK parser. Pin a custom error page / response that matches the envelope so a gateway `502/504`
still parses:

```nginx
# nginx — make gateway errors speak problem+json, not HTML
error_page 502 503 504 = @upstream_down;
location @upstream_down {
    default_type application/problem+json;
    return 503 '{"type":"https://errors.dataquery.example/upstream-unavailable","title":"Upstream unavailable","status":503,"detail":"The data service is temporarily unreachable.","error":{"code":"upstream_unavailable","message":"The data service is temporarily unreachable.","retryable":true,"requestId":"req_gateway"}}';
    add_header Retry-After 5 always;
}
```

Envoy's `local_reply` config and Cloudflare Workers' `Response` give the equivalent hook. The principle:
**no layer in the request path is allowed to emit a body the SDK can't parse.** The catch-all handler
(§9.2) covers the app; this covers everything in front of it.

### 10.2 The streaming / SSE channel equivalent

A long batch export may stream NDJSON or SSE. You cannot change the HTTP status after the first byte is
sent (it's already `200`). So mid-stream failures are carried **in-band** as a terminal error frame —
the same envelope, just framed:

```python
# SSE: a per-row failure or a terminal error are events, not status codes
async def stream_batch(exprs: list[str]) -> AsyncIterator[bytes]:
    yield b'event: meta\ndata: {"requestId":"req_..."}\n\n'
    for expr in exprs:
        res = await fetch_one(expr, provider_for(expr))
        yield f'event: series\ndata: {res.model_dump_json()}\n\n'.encode()  # status per row (§4.3)
    yield b'event: done\ndata: {"summary":{"ok":1,"unavailable":1}}\n\n'

# A FATAL mid-stream error (store died) → a terminal error EVENT, same code vocabulary:
# event: error
# data: {"error":{"code":"store_unavailable","message":"...","retryable":true,"requestId":"req_..."}}
```

The SDK's streaming reader maps an `event: error` frame onto the *same* `_BY_CODE` table (§7.2) and raises
the *same* typed exception it would for a non-streamed `503`. One vocabulary, three transports (body,
gateway, stream). This mirrors Lumina's own non-negotiable #5 ("stream → wire tail → persist before
`res.end()`") on the read side: the failure state is part of the wire protocol, not an afterthought.

## 11. Anti-patterns (mistake → fix)

| Mistake | Why it breaks | Fix |
|---|---|---|
| A failed provider fetch returns a `0`, `null`, last-known, or zero-filled point to "look complete". | Violates non-negotiable #1 — a fabricated/backfilled finance number. Indistinguishable downstream from a real value. | Return `SeriesStatus.unavailable` with `points: []`. The type makes a non-empty non-`ok` row impossible (§4.3). |
| Missing provider key → HTTP `500`. | A configuration fact pages on-call as a "crash"; the SDK can't branch to "needs a key". | `needsKey` per-series data (or `200` `{status:needsKey, series:[]}` for a whole keyless dataset). §4. |
| One bad ticker fails the whole 10-series batch (`400`/`503`). | Forces the client to discard 9 good series → pressures it to backfill the missing one. | Per-series status; batch stays `200` unless *all* failed & ≥1 transient (§5.2). |
| `license_denied` modeled as `unavailable` (or vice-versa). | Hides the fix: "purchase a license" vs "retry later" are opposite actions. Licensing is orthogonal to availability (`commercial-ok-gate.md`). | `license_denied` ⇒ `403` (or per-series `license_denied`); `unavailable` ⇒ data/`503`. §4.2. |
| `commercialOk` defaults to `true` / omitted. | A RED free-tier number ships as redistributable. The gate's default is **false**. | `Provenance.commercialOk: bool = False`; only `true` with a 🟢 ledger row. |
| SDK keys exceptions off the **HTTP status** (`if status==400`). | RFC 9457 §3 forbids it — many codes share a status; the SDK can't tell `range_inverted` from `too_many_expressions`. | Key off `error.code` (§7.2). Status is a coarse hint only. |
| `error.code` renamed/repurposed in a release. | Silently breaks every `except SomeError` in every consumer — a breaking change with no compile error. | Codes are a frozen public enum: add, never rename (§3.2). |
| `detail`/`message` leaks a stack trace, SQL, or a secret. | RFC 9457 §3.1: detail should help correct the problem, not debug it. Leaks attack surface. | Generic safe message in the body; full traceback to logs keyed by `requestId` (§9.2 catch-all). |
| 422 returned in FastAPI's raw `{"detail":[{loc,msg,type}]}` shape. | Doesn't match the envelope; SDK has two shapes to parse; no `code`/`retryable`. | Override `RequestValidationError` → reshape into the envelope (§9.2). |
| `429`/`503` with no `Retry-After`. | Clients hammer with no backoff signal → thundering herd. | Always send `Retry-After` (seconds) on 429; SHOULD on 503. Echo in `details`. §8.1. |
| Idempotency-Key cache stored in process memory on a multi-instance deploy. | Two instances each execute the write → duplicate job. | Redis/Postgres with `SET … NX EX 86400` + a lock; replay the saved status+body. §6.2. |
| Same idempotency key + different params silently runs again. | A client bug becomes a duplicate side-effect. | `409 idempotency_conflict` on fingerprint mismatch (Stripe rule). §6.1. |
| The OpenAPI doc declares `application/problem+json` for errors but no `code` enum / per-status response. | Generator emits one generic, unhelpful exception (OpenAPI-Generator #20742, Kiota #4349). | Enumerate `code` in the schema + one `responses` component per status (§7.1). |
| A gateway `502`/`504` returns HTML. | Breaks the SDK's JSON parser before the app is even reached. | Pin a `problem+json` error page at the gateway (§10.1). |
| Mid-stream failure tries to change the HTTP status. | Status is already `200`; bytes are sent. | Terminal `event: error` frame carrying the same envelope (§10.2). |

## 12. Grading rubric (does this contract pass review?)

A delivery channel's error handling is **done** only when all of the following are true:

1. **One envelope.** Every channel (REST body, gateway, stream) emits the *same* RFC-9457-based shape with
   the *same* `error.{code, message, retryable, requestId, details?}` extension. No endpoint returns a
   bare `{"detail": …}` or HTML.
2. **`type` URI + stable `code`.** Every error carries an RFC 9457 `type` and a machine `code` from the
   frozen enum; the SDK keys exceptions off `code`, never off the bare status (RFC 9457 §3).
3. **Status map obeyed.** 400/401/403/404/409/422/429/503 each map to the documented code(s) per §8; no
   expected failure is a `500`; the catch-all wraps unhandled bugs as `code:"internal"`, no traceback in
   the body.
4. **`needsKey`/`unavailable`/`license_denied` are first-class and distinct.** A missing key, a failed
   fetch, and an unlicensed series are three different typed states, never collapsed, never a `500`, and
   never a fabricated/backfilled number (`points:[]` invariant; non-negotiable #1).
5. **Partial success works.** A multi-series request reports a status per series; one failure does not
   fail the others; the batch HTTP status follows the §5.2 rule (200 unless all-failed-and-transient).
6. **`commercialOk` defaults false; `license_denied`→403.** No RED series ships as redistributable; the
   licensing state is its own code, separate from availability.
7. **`retryable` is correct per code, and the SDK acts on it.** 429/503 retryable with `Retry-After`
   honored + jittered backoff fallback; 400/401/403/404/409/422 not blind-retried; auth_expired and
   in-flight-conflict route to a corrective action, not a blind retry.
8. **Idempotency is real.** Mutating endpoints accept `Idempotency-Key`; same-key+same-params replays the
   saved status+body (incl. errors); same-key+different-params → `409`; store is distributed + 24h TTL +
   locked.
9. **OpenAPI documents it.** One reusable `ProblemDetails` schema with the `code` enum, one `responses`
   component per status, referenced by every operation, so the generated SDK produces typed exceptions
   (no generic-exception failure mode).
10. **Every error correlates.** A `requestId` is in the body, in the `X-Request-Id` response header, and
    in the server log line for the failure.

If any row is false, the channel is **not** done — a consumer somewhere will have to special-case an
undocumented failure shape, which is the exact failure this contract exists to prevent.
