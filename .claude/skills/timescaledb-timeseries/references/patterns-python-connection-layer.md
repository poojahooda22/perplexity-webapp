# patterns-python-connection-layer.md

> **Product line:** JPM-Markets re-engineering **data-analytics** product line (a NEW Python/FastAPI
> backend, **NOT** Lumina's Bun/Express app). This recipe is the connection + query layer between a
> FastAPI service and a TimescaleDB (PostgreSQL + the `timescaledb` extension) database. It is the
> Python equivalent of Lumina's `cache.ts`/`tools.ts` split: **the route owns HTTP, this layer owns the
> SQL.** Greenfield — there are no codebase `file:line` anchors yet; everything below is a build recipe
> grounded in primary docs.

**What this doc decides for you, up front (the verdicts):**

1. **Driver:** use **`asyncpg`** as the wire driver for the read/write API path. It is the fastest async
   PostgreSQL driver and the canonical async dialect for SQLAlchemy. Use **`psycopg3`** only where you
   want one driver that does sync *and* async, or richer COPY ergonomics. Do **not** use `psycopg2`
   (sync-only, no asyncio) on a FastAPI async path.
2. **ORM vs raw:** drive **time-series analytical queries as raw parameterized SQL** through a thin
   repository module — not the ORM. Use SQLAlchemy Core/ORM (optional) for *relational metadata* tables
   (instruments, users, saved views). Hyperfunctions (`time_bucket`, `first`, `last`, `candlestick_agg`)
   are just SQL; the ORM gives you nothing here and its identity map/lazy-load machinery actively gets in
   the way of streaming wide result sets.
3. **Hypertable DDL** (`create_hypertable`, compression/retention policies, continuous aggregates) lives
   in **Alembic migrations as `op.execute("SELECT …")`** — raw SQL, never autogenerated. The ORM model
   maps the hypertable as an ordinary table; Timescale's chunk machinery is invisible to it.
4. **Pooling:** one long-lived `asyncpg.Pool` created in the FastAPI **lifespan**, stored on
   `app.state`. If (and only if) a **transaction-mode pgbouncer/Supavisor** sits between the app and
   Postgres, disable the prepared-statement cache (`statement_cache_size=0`) — otherwise you get
   intermittent `prepared statement "__asyncpg_stmt_xx__" does not exist` errors.
5. **Bulk ingest:** use **`copy_records_to_table`** (asyncpg) / `cursor.copy()` (psycopg3) — the
   `COPY` protocol — for batch loads. Never `INSERT … executemany` for large batches.

## 1. The layering law — where SQL lives <a name="1-the-layering-law"></a>

Lumina's `skill-layer-law.md` separates three things: a **route** (owns HTTP), a **tool/fetcher** (owns
the actual data fetch), and **reasoning** (the agent). This Python product line gets the same discipline,
mapped to FastAPI:

| Layer | File | One-line test | Mirror in Lumina |
|---|---|---|---|
| **Route** | `app/api/routers/prices.py` | "Does this parse the request, call a service, and shape the HTTP response?" | a `routes.ts` route |
| **Service / repository** | `app/repositories/ohlcv_repo.py` | "Does this **build and run the SQL** against TimescaleDB and return typed rows?" | `tools.ts` fetcher |
| **Domain / schema** | `app/schemas/ohlcv.py` (Pydantic), `app/models/*.py` (SQLAlchemy) | "Does this define the shape of data crossing the boundary?" | `Provenance`/types |
| **DB session/pool** | `app/db/pool.py` | "Does this own the connection lifecycle?" | the cache/client singletons |

**The rule that bites:** a FastAPI route must **never contain a SQL string**. If you find an
`f"SELECT ... {table}"` inside a path operation function, it is in the wrong layer — exactly as fetch
logic in a Lumina runtime-skill "never runs." The route's body should read like English:

```python
# app/api/routers/prices.py  — the ROUTE owns HTTP only
from fastapi import APIRouter, Depends, Query
from datetime import datetime
from app.repositories.ohlcv_repo import OhlcvRepository, get_ohlcv_repo
from app.schemas.ohlcv import OhlcvBar

router = APIRouter(prefix="/v1/prices", tags=["prices"])

@router.get("/{symbol}/ohlcv", response_model=list[OhlcvBar])
async def get_ohlcv(
    symbol: str,
    start: datetime = Query(...),
    end: datetime = Query(...),
    bucket: str = Query("1 hour"),
    repo: OhlcvRepository = Depends(get_ohlcv_repo),
) -> list[OhlcvBar]:
    # No SQL here. The route validates inputs and delegates.
    return await repo.fetch_ohlcv(symbol=symbol, start=start, end=end, bucket=bucket)
```

Everything in the rest of this doc is *the service layer* — the SQL home. The route above is the only
HTTP-aware code; the repository (§10) is the only SQL-aware code.

## 2. Driver selection: asyncpg vs psycopg3 vs SQLAlchemy <a name="2-driver-selection"></a>

There are three real choices for talking to Postgres from async Python. They are not interchangeable.

### 2.1 The decision table

| Need | Pick | Why |
|---|---|---|
| Fastest async wire path; raw SQL; high-throughput time-series reads | **asyncpg** | Binary protocol, server-side prepared statements, C-accelerated row decode. "asyncpg is, on average, 5x faster than psycopg3" per [PyPI/benchmarks](https://pypi.org/project/asyncpg/). |
| One driver for both sync scripts (cron ingest, Alembic) and async API | **psycopg3** | Single codebase does `psycopg.connect()` (sync) and `await psycopg.AsyncConnection.connect()` (async). |
| You want an ORM / query builder, migrations autogen, relational metadata modeling | **SQLAlchemy 2.0** (over asyncpg) | `create_async_engine("postgresql+asyncpg://...")`. The ORM rides on top of asyncpg. |
| Sync-only legacy, no asyncio | psycopg2 | **Avoid on FastAPI** — it blocks the event loop. |

**Recommended split for this product line:**

- **API read/write path** → `asyncpg` directly via a repository, *or* SQLAlchemy-async if the team wants
  the ORM for the relational side. Pick one and be consistent per service.
- **Alembic migrations** → SQLAlchemy + a **sync** driver (`psycopg`/`psycopg2`) is simplest; Alembic can
  run async too, but the sync engine in migrations avoids event-loop plumbing in the migration env.
- **Batch ingest worker** (the nightly/streaming loader) → `asyncpg.copy_records_to_table` or psycopg3
  `cursor.copy()`.

### 2.2 Why not just always use the ORM?

For **time-series analytical queries**, the ORM is a liability, not a help:

1. **Hyperfunctions are SQL the ORM doesn't model.** `time_bucket('1 hour', ts)`, `candlestick_agg(...)`,
   `last(price, ts)` have no ORM expression. You'd write `text("...")` anyway — at which point the ORM is
   pure overhead.
2. **Wide aggregate result sets** (10k buckets × N series) don't want the ORM identity map or object
   hydration. You want plain tuples/dicts → Pydantic.
3. **Lazy loading is a footgun in async.** In SQLAlchemy async, touching an un-loaded relationship raises
   unless you used `selectinload()` or `AsyncAttrs.awaitable_attrs`
   ([SQLAlchemy asyncio docs](https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html)). Time-series
   rows have no relationships to lazy-load, so the ORM buys nothing and adds a sharp edge.

**Verdict:** ORM for *metadata* (instruments, dashboards, saved queries — rows you mutate and relate),
raw SQL for *measurements* (the hypertable reads). Same database, two access styles, chosen per table by
what the table is for.

## 3. Connection pooling with asyncpg <a name="3-connection-pooling-with-asyncpg"></a>

### 3.1 Why pool at all

Opening a PostgreSQL connection is expensive (TCP + TLS + auth + backend fork). Under any real request
rate you must reuse connections. asyncpg ships its own pool — `asyncpg.create_pool()` — and the asyncpg
maintainers explicitly recommend it over pgbouncer when pgbouncer's only job is connection reuse: *"if you
are using pgbouncer only to reduce the cost of new connections … switch to the connection pool
functionality provided by asyncpg, it is a much better option for this purpose"*
([asyncpg FAQ](https://magicstack.github.io/asyncpg/current/faq.html)).

### 3.2 `create_pool()` parameters that matter

From the [asyncpg API reference](https://magicstack.github.io/asyncpg/current/api/index.html):

| Param | Default | What it controls |
|---|---|---|
| `min_size` | `10` | Connections the pool is initialized with (pre-warmed). |
| `max_size` | `10` | Hard cap on concurrent connections. |
| `max_queries` | `50000` | After this many queries a connection is closed and replaced (defends against server-side memory creep / stale plans). |
| `max_inactive_connection_lifetime` | `300.0` s | Idle connections older than this are closed. |
| `statement_cache_size` | `100` | Size of the per-connection prepared-statement LRU. **Set to `0` behind a transaction pooler** (§5). |
| `max_cached_statement_lifetime` | `300` s | TTL for a cached prepared statement. |
| `command_timeout` | `None` | Default per-operation timeout (seconds). |
| `server_settings` | `None` | dict of Postgres GUCs applied per connection (e.g. `{"application_name": "analytics-api", "timezone": "UTC"}`). |
| `init` / `setup` | `None` | Coroutines run once per new connection / each acquire — register custom type codecs here. |

> **Important default trap:** asyncpg defaults `min_size=10, max_size=10`. That means the pool opens **10
> connections immediately** and never grows beyond 10. For a small service that is fine; for a fleet of
> pods behind a shared Postgres, `10 pods × 10 = 100` connections — size deliberately against your
> `max_connections` (default 100 on stock Postgres) and any pooler limits.

### 3.3 Creating the pool

```python
# app/db/pool.py
from __future__ import annotations
import asyncpg
from app.config import settings

async def create_pool() -> asyncpg.Pool:
    return await asyncpg.create_pool(
        dsn=settings.database_url,            # postgresql://user:pass@host:5432/analytics
        min_size=settings.db_pool_min,        # e.g. 2
        max_size=settings.db_pool_max,        # e.g. 10  (sized vs Postgres max_connections)
        max_queries=50_000,
        max_inactive_connection_lifetime=300.0,
        command_timeout=30.0,                 # fail slow queries rather than hang a request
        server_settings={
            "application_name": "analytics-api",
            "timezone": "UTC",                # store & compute in UTC; convert at the edge
            "jit": "off",                     # JIT rarely helps short OLAP-ish reads; reduces tail latency
        },
        # statement_cache_size=0,             # uncomment ONLY behind a transaction-mode pooler (§5)
    )
```

### 3.4 Acquiring and using connections

`Pool.acquire()` returns a `Connection` and "can be used in an `await` expression or with an `async with`
block" ([API ref](https://magicstack.github.io/asyncpg/current/api/index.html)). **Always use
`async with`** so the connection is returned even on exception:

```python
async with pool.acquire() as conn:        # checked out
    rows = await conn.fetch("SELECT ...")  # conn is an asyncpg.Connection
# conn automatically released back to the pool here
```

The pool also exposes convenience methods that acquire+release for you and "behave identically to their
`Connection` equivalents" ([API ref](https://magicstack.github.io/asyncpg/current/api/index.html)):

```python
rows  = await pool.fetch("SELECT ...", arg1)      # acquire → fetch → release, one call
row   = await pool.fetchrow("SELECT ... LIMIT 1", arg1)
value = await pool.fetchval("SELECT count(*) FROM trades WHERE symbol=$1", symbol)
await pool.execute("UPDATE ...", arg1)
```

Use the pool convenience methods for **single statements**; use `async with pool.acquire()` when you need
a **transaction** or **multiple statements on the same connection** (e.g. a `COPY` then a `SELECT` that
must see it, or `SET LOCAL` + query):

```python
async with pool.acquire() as conn:
    async with conn.transaction():        # BEGIN … COMMIT (ROLLBACK on exception)
        await conn.execute("SET LOCAL statement_timeout = '5s'")
        await conn.copy_records_to_table("staging_ticks", records=batch, columns=cols)
        await conn.execute("INSERT INTO ticks SELECT * FROM staging_ticks ON CONFLICT DO NOTHING")
```

### 3.5 Type codecs (custom decoding)

asyncpg returns native Python types for most Postgres types, but for things like JSON you typically
register codecs in the pool `init` so every connection decodes consistently:

```python
import json

async def _init_conn(conn: asyncpg.Connection) -> None:
    await conn.set_type_codec(
        "jsonb",
        encoder=json.dumps,
        decoder=json.loads,
        schema="pg_catalog",
    )

pool = await asyncpg.create_pool(dsn=..., init=_init_conn)
```

This is the right place to register a `numeric` → `Decimal`/`float` policy too, if you want floats for
analytics throughput instead of `Decimal` (asyncpg returns `Decimal` for `numeric` by default, which is
*correct for money* — don't blindly cast prices to float; see §9).

## 4. The FastAPI lifespan + dependency wiring <a name="4-the-fastapi-lifespan"></a>

The pool must be created **once at startup** and closed at shutdown — not per request. FastAPI's modern
mechanism is the **lifespan context manager** (`@asynccontextmanager`), which replaced the deprecated
`@app.on_event("startup")`. Code before `yield` runs once on startup; code after runs once on shutdown
([FastAPI lifespan pattern](https://daniel.feldroy.com/posts/2025-10-using-asyncpg-with-fastapi-and-air)).

### 4.1 The canonical wiring

```python
# app/main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI
from app.db.pool import create_pool
from app.api.routers import prices, series

@asynccontextmanager
async def lifespan(app: FastAPI):
    # --- startup: runs ONCE ---
    app.state.pool = await create_pool()
    yield
    # --- shutdown: runs ONCE ---
    await app.state.pool.close()   # graceful: waits for in-flight queries, then closes all conns

app = FastAPI(lifespan=lifespan, title="JPM-Markets Analytics API")
app.include_router(prices.router)
app.include_router(series.router)
```

> `pool.close()` is graceful (drains in-flight work). For a force-stop on a stuck shutdown you can wrap it:
> `await asyncio.wait_for(app.state.pool.close(), timeout=10)` and fall back to `pool.terminate()`.

### 4.2 The dependency that hands the pool/repo to routes

Never reach into `app.state` from inside a route by importing the app (circular). Use a FastAPI
dependency that reads it off the `Request`:

```python
# app/db/deps.py
from fastapi import Request
import asyncpg

def get_pool(request: Request) -> asyncpg.Pool:
    return request.app.state.pool
```

```python
# app/repositories/ohlcv_repo.py  (the constructor side)
from fastapi import Depends
import asyncpg
from app.db.deps import get_pool

class OhlcvRepository:
    def __init__(self, pool: asyncpg.Pool) -> None:
        self._pool = pool
    # ... methods in §10 ...

def get_ohlcv_repo(pool: asyncpg.Pool = Depends(get_pool)) -> OhlcvRepository:
    return OhlcvRepository(pool)
```

Now a route declares `repo: OhlcvRepository = Depends(get_ohlcv_repo)` and never touches the pool, the
connection, or SQL directly. This is the FastAPI translation of "secure tool args by closure" — the route
can't fabricate a connection; it receives a repository bound to the one true pool.

## 5. The pgbouncer / transaction-pooler prepared-statement trap <a name="5-the-pgbouncer-trap"></a>

This is the single highest-blast-radius gotcha in the whole connection layer. Get it wrong and you ship a
service that works in staging and throws `prepared statement "__asyncpg_stmt_3__" does not exist` under
load in production.

### 5.1 The mechanism

asyncpg, for speed, uses **server-side prepared statements** by default (`statement_cache_size=100`). A
prepared statement is created on a *specific Postgres backend connection* and referenced by name on later
executions.

A **transaction-mode** pooler (pgbouncer `pool_mode=transaction`, Supabase Supavisor transaction pooler)
multiplexes many client connections over few server connections, **handing you a different backend per
transaction**. So the connection that *prepared* `__asyncpg_stmt_3__` is not the connection you *execute*
it on next time → `prepared statement … does not exist` (or `DuplicatePreparedStatementError` if two
clients collide on the same name). The asyncpg FAQ states this plainly: *"pgbouncer, when in the
`transaction` or `statement` pooling mode, does not support prepared statements"*
([asyncpg FAQ](https://magicstack.github.io/asyncpg/current/faq.html)).

### 5.2 The fix matrix

| Your topology | Action |
|---|---|
| App → **direct** Postgres (no pooler) | Nothing. Keep prepared statements ON (the fast default). Use asyncpg's own pool. |
| App → pgbouncer **`pool_mode=session`** | Nothing. Session mode is compatible with prepared statements; asyncpg recommends switching pgbouncer to `session` if you keep it ([asyncpg FAQ](https://magicstack.github.io/asyncpg/current/faq.html)). |
| App → pgbouncer/Supavisor **transaction mode** | **Disable the statement cache** (below). |

### 5.3 Disabling the statement cache — raw asyncpg

Pass `statement_cache_size=0` to **both** `connect()` and `create_pool()`
([asyncpg FAQ](https://magicstack.github.io/asyncpg/current/faq.html)). It is **not** settable via the DSN
query string — it must be a kwarg:

```python
pool = await asyncpg.create_pool(
    dsn=settings.database_url,
    statement_cache_size=0,   # no server-side prepared statements → transaction-pooler safe
)
```

And avoid `Connection.prepare()` (explicit prepared statements) on that path.

### 5.4 Disabling it — SQLAlchemy + asyncpg

SQLAlchemy's asyncpg dialect has **its own** prepared-statement cache, separate from asyncpg's, *because
SQLAlchemy calls `asyncpg.connection.prepare()` itself*. So behind a transaction pooler you must set
**both**:

```python
from sqlalchemy.ext.asyncio import create_async_engine
from sqlalchemy.pool import NullPool

engine = create_async_engine(
    "postgresql+asyncpg://user:pass@host/db?prepared_statement_cache_size=0",
    poolclass=NullPool,                              # don't double-pool on top of the external pooler
    connect_args={"statement_cache_size": 0},        # asyncpg's own cache OFF too
)
```

([SQLAlchemy discussion #10246](https://github.com/sqlalchemy/sqlalchemy/discussions/10246) — note the
URL query arg `?prepared_statement_cache_size=0` *and* the `connect_args` both appear in the recommended
config.)

If multiple app instances share a pooler and you still hit `DuplicatePreparedStatementError` from name
collisions, give each prepared statement a unique name. The modern, preferred form uses **unnamed
prepared statements** (empty name) so there is nothing to collide on:

```python
engine = create_async_engine(
    dsn,
    connect_args={
        "prepared_statement_name_func": lambda: "",   # unnamed → no collision, no leak
        "statement_cache_size": 0,
    },
    poolclass=NullPool,
)
```

The older workaround is a UUID-suffixed name via a custom `Connection` subclass
([SQLAlchemy discussion #10246](https://github.com/sqlalchemy/sqlalchemy/discussions/10246)):

```python
from uuid import uuid4
from asyncpg import Connection

class CConnection(Connection):
    def _get_unique_id(self, prefix: str) -> str:
        return f"__asyncpg_{prefix}_{uuid4()}__"

engine = create_async_engine(dsn, connect_args={"connection_class": CConnection})
```

### 5.5 The cost — and the senior call

SQLAlchemy maintainer Mike Bayer warns that `prepared_statement_cache_size=0` *"definitely degrades
performance very measurably"* and recommends evaluating **session-mode pooling** instead
([SQLAlchemy discussion #10246](https://github.com/sqlalchemy/sqlalchemy/discussions/10246)). So the
ranked preference for this product line is:

1. **Direct Postgres + asyncpg's own pool** — fastest, prepared statements ON. (Default choice for a
   long-lived FastAPI worker on a dedicated DB — §6.)
2. **Session-mode pooler** if you need one — prepared statements still work.
3. **Transaction-mode pooler + caches disabled** — only when forced (serverless, Supabase's transaction
   pooler), and accept the measured slowdown.

Document which one you chose in the service README and *test against the real topology* — the bug is
invisible against a direct local Postgres.

## 6. Serverless vs long-lived worker pooling <a name="6-serverless-vs-long-lived"></a>

Lumina's non-negotiable #4 ("Vercel can't hold sockets or timers") has a direct analogue here: **where the
process lives determines how you pool.**

### 6.1 Long-lived worker (recommended for this product line)

The data-analytics API should run as a **persistent ASGI process** (uvicorn/gunicorn-uvicorn workers on a
container — Fly.io / a VM / Kubernetes), exactly like Lumina's `worker/`. A persistent process:

- Creates **one `asyncpg.Pool` in lifespan** and reuses it for the process lifetime.
- Keeps prepared statements ON (fastest).
- Sizes the pool against `max_connections`: `pool_max × num_workers × num_pods ≤ Postgres max_connections`
  (leave headroom for migrations/admin).

This is the default. TimescaleDB analytical workloads (range scans over chunks, continuous-aggregate
reads) reward warm, prepared, long-lived connections.

### 6.2 Serverless (if a route ever runs on Lambda/Cloud Run scale-to-zero)

A function instance is created and frozen per request burst; it **cannot keep a 10-connection pool warm**,
and a swarm of cold instances will exhaust `max_connections`. If you must run serverless:

- Put a **transaction-mode pooler (pgbouncer/Supavisor/RDS Proxy)** in front, and disable prepared
  statements per §5.
- Use a **tiny pool** per instance (`min_size=0, max_size=1–2`) or a fresh `connect()` per invocation, and
  **dispose** it before the function returns. With SQLAlchemy use `poolclass=NullPool` so SQLAlchemy
  doesn't try to keep its own pool across frozen invocations
  ([SQLAlchemy asyncio docs](https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html) — `NullPool`
  + `AsyncEngine.dispose()` is the documented multi-loop/serverless guidance).

```python
# serverless handler shape (last resort)
import asyncpg
async def handler(event):
    conn = await asyncpg.connect(dsn=POOLER_DSN, statement_cache_size=0)  # transaction pooler in front
    try:
        return await conn.fetch("SELECT ...")
    finally:
        await conn.close()
```

> **Senior call:** heavy time-series analytics on a serverless request path is the wrong shape — the cold
> pool + pooler + disabled-prepared-statement tax stacks up. Run the analytics API as a **long-lived
> worker** (§6.1) and reserve serverless for thin, low-QPS endpoints. This mirrors Lumina putting sockets
> and pollers in `worker/`, not on the serverless path.

## 7. Calling Timescale functions & hyperfunctions through the driver <a name="7-calling-timescale-functions"></a>

**The whole point:** TimescaleDB adds no special client protocol. `create_hypertable`, `time_bucket`,
`first`/`last`, `candlestick_agg`, `add_compression_policy`, continuous-aggregate refresh — **all of it is
just SQL functions.** You call them through asyncpg/psycopg/SQLAlchemy exactly like any `SELECT`. There is
no "Timescale driver." This is liberating: every hyperfunction is reachable from day one with the plain
driver you already have.

### 7.1 A `time_bucket` read through asyncpg

```python
async def fetch_hourly_volume(pool, symbol: str, start, end) -> list[dict]:
    rows = await pool.fetch(
        """
        SELECT time_bucket('1 hour', ts)        AS bucket,
               sum(volume)                        AS total_volume
        FROM   trades
        WHERE  symbol = $1
          AND  ts >= $2
          AND  ts <  $3
        GROUP  BY bucket
        ORDER  BY bucket
        """,
        symbol, start, end,
    )
    return [dict(r) for r in rows]   # asyncpg.Record → dict
```

`$1/$2/$3` are asyncpg's positional placeholders (psycopg uses `%s`; see §14). Nothing about
`time_bucket` needs special handling — it returns a `timestamptz`, asyncpg decodes it to a Python
`datetime`.

### 7.2 An OHLCV bar via the financial-analytics hyperfunctions

TimescaleDB's Toolkit ships `candlestick_agg` (and accessors `open`, `high`, `low`, `close`, `volume`).
These are SQL functions too:

```python
OHLCV_SQL = """
SELECT time_bucket($1::interval, ts)            AS bucket,
       open(candlestick_agg(ts, price, volume))  AS open,
       high(candlestick_agg(ts, price, volume))  AS high,
       low(candlestick_agg(ts, price, volume))   AS low,
       close(candlestick_agg(ts, price, volume)) AS close,
       volume(candlestick_agg(ts, price, volume))AS volume
FROM   ticks
WHERE  symbol = $2 AND ts >= $3 AND ts < $4
GROUP  BY bucket
ORDER  BY bucket
"""
rows = await pool.fetch(OHLCV_SQL, bucket, symbol, start, end)
```

(If you don't use Toolkit, the equivalent is `first(price, ts)`, `last(price, ts)`, `max`, `min`,
`sum(volume)` — also plain SQL.)

### 7.3 Administrative Timescale functions are just `execute`

```python
# Refresh a continuous aggregate on demand (e.g. after a backfill) — note: NOT inside a txn block
await pool.execute(
    "CALL refresh_continuous_aggregate('ohlcv_1h', $1, $2)", start, end
)
```

> `refresh_continuous_aggregate` is a **procedure** (`CALL`), and it cannot run inside an explicit
> transaction — so call it via `pool.execute`/a fresh connection, **not** inside `async with
> conn.transaction()`. This is a Timescale constraint, surfaced through the driver as a normal error if
> you wrap it.

**Rule:** policies, hypertable creation, compression, and continuous-aggregate *definitions* belong in
**migrations** (§13), not runtime code. Only *reads* and on-demand *refreshes* belong in the repository.

## 8. Parameterizing time-range queries safely <a name="8-parameterizing-time-range-queries"></a>

Time-range queries are the bread and butter of this layer, and they are where SQL-injection and full-table
scans both hide.

### 8.1 Never f-string a value into SQL

```python
# WRONG — SQL injection + no plan caching
sql = f"SELECT * FROM trades WHERE symbol = '{symbol}' AND ts >= '{start}'"

# RIGHT — bound parameters (asyncpg positional)
rows = await pool.fetch(
    "SELECT * FROM trades WHERE symbol = $1 AND ts >= $2 AND ts < $3",
    symbol, start, end,
)
```

Bound parameters are passed out-of-band over the wire — the value can never be parsed as SQL. asyncpg
binds Python `datetime` objects to `timestamptz` natively, so **pass real `datetime`s, not strings.**

### 8.2 The identifier problem — bucket width, table name, column

You **cannot** bind an *identifier* or a *keyword* (table name, column name, `ASC`/`DESC`) as a parameter
— only values. For a user-chosen bucket like `'1 hour'`, two safe options:

**Option A — bind it as an `interval` value** (preferred, since a bucket width *is* a value):

```python
# bucket arrives as a string like "1 hour"; bind it, cast in SQL
rows = await pool.fetch(
    "SELECT time_bucket($1::interval, ts) AS bucket, sum(volume) "
    "FROM trades WHERE symbol=$2 AND ts>=$3 AND ts<$4 GROUP BY bucket ORDER BY bucket",
    bucket, symbol, start, end,
)
```

This is safe: `$1::interval` is a *value cast*, and an invalid string just raises a clean Postgres error
(`invalid input syntax for type interval`) — no injection.

**Option B — allow-list identifiers** when you genuinely need a dynamic column/table/direction:

```python
ALLOWED_SORT = {"bucket", "total_volume"}            # closed set, never user-free-text
ALLOWED_DIR  = {"ASC", "DESC"}

def build_order(col: str, direction: str) -> str:
    if col not in ALLOWED_SORT or direction.upper() not in ALLOWED_DIR:
        raise ValueError("bad sort spec")
    return f"ORDER BY {col} {direction.upper()}"     # safe: both came from a closed allow-list
```

The principle: **values → bound parameters; identifiers → allow-listed enums.** There is no third option.
(If you ever need many dynamic identifiers, psycopg3's `sql.Identifier`/`sql.SQL` composition is the
typed, injection-safe builder — see §14.4.)

### 8.3 Validate the range and cap the span at the boundary (Pydantic)

A time-range endpoint is a **scale surface** (R-SCALE): an unbounded `start..end` over a 1M+ row
hypertable is a denial-of-service on yourself. Cap the span and require `start < end` in the request model
so a bad range never reaches SQL:

```python
# app/schemas/query.py
from datetime import datetime, timedelta
from pydantic import BaseModel, model_validator

MAX_SPAN = timedelta(days=370)        # one year + buffer; tune per resolution

class TimeRange(BaseModel):
    start: datetime
    end: datetime

    @model_validator(mode="after")
    def _check(self) -> "TimeRange":
        if self.start >= self.end:
            raise ValueError("start must be before end")
        if self.end - self.start > MAX_SPAN:
            raise ValueError(f"range too large; max {MAX_SPAN.days} days")
        return self
```

FastAPI turns a `ValueError` in a Pydantic model into a **422** automatically — the route never runs, the
DB is never touched. This is the "validate at the trust boundary" rule.

### 8.4 Index discipline (the R-SCALE bite)

Bound parameters don't save you from a full scan. The hypertable's reads must be **chunk-excludable**:
filter on the **partitioning time column** (so Timescale prunes chunks) and have a composite index on the
columns you actually filter:

```sql
-- created in a migration, not at runtime
CREATE INDEX ON trades (symbol, ts DESC);
```

A query that filters `WHERE symbol = $1 AND ts >= $2 AND ts < $3` then prunes chunks by `ts` and uses the
`(symbol, ts)` index inside the surviving chunks. A query that filters on an **unindexed** column is a
full scan of every chunk — the "reading every page of a book instead of the index" failure from R-SCALE.
State, per endpoint, which index serves it.

## 9. Typed result mapping: rows → Pydantic → FastAPI response <a name="9-typed-result-mapping"></a>

The repository returns **typed Pydantic models**, never raw `asyncpg.Record`s, so the FastAPI
`response_model` can validate and serialize, and the OpenAPI schema is correct.

### 9.1 Define the response schema

```python
# app/schemas/ohlcv.py
from datetime import datetime
from decimal import Decimal
from pydantic import BaseModel, ConfigDict

class OhlcvBar(BaseModel):
    model_config = ConfigDict(from_attributes=True)  # allow building from objects w/ attrs

    bucket: datetime
    open: Decimal        # money: keep Decimal end-to-end (see 9.3)
    high: Decimal
    low: Decimal
    close: Decimal
    volume: Decimal
```

### 9.2 Map asyncpg `Record` → Pydantic

`asyncpg.Record` supports both index and **key** access and is mapping-like, so `dict(record)` or
`Model(**record)` works. Validate explicitly:

```python
rows = await pool.fetch(OHLCV_SQL, bucket, symbol, start, end)
return [OhlcvBar.model_validate(dict(r)) for r in rows]
```

`model_validate(dict(r))` (Pydantic v2) coerces and **validates** every field — a NULL where the schema
says non-optional raises here, in the service, not as a 500 later in serialization. That validation step
is the typed boundary; don't skip it by returning bare dicts.

### 9.3 The money decode — do NOT carelessly cast to float

asyncpg decodes Postgres `numeric` to Python **`Decimal`** by default. **Keep it.** A price stored as
`numeric(18,8)` must not become a binary `float` (which can't represent `0.1` exactly) on the way to the
client — that silently corrupts money, exactly the class of error Lumina's non-negotiable #1 ("never
invent a finance number") guards against. Pydantic serializes `Decimal` to a JSON number/string
faithfully. Only convert to `float` for charting math where the precision loss is acceptable and
documented, and never for displayed prices or P&L.

### 9.4 Streaming large result sets

For an export endpoint returning hundreds of thousands of bars, don't materialize the whole list. Use a
**server-side cursor** (asyncpg `Connection.cursor()` inside a transaction) and a FastAPI
`StreamingResponse`:

```python
from fastapi.responses import StreamingResponse

async def stream_ticks(pool, symbol, start, end):
    async def gen():
        async with pool.acquire() as conn:
            async with conn.transaction():            # cursor requires a transaction
                async for r in conn.cursor(
                    "SELECT ts, price, volume FROM ticks "
                    "WHERE symbol=$1 AND ts>=$2 AND ts<$3 ORDER BY ts",
                    symbol, start, end,
                ):
                    yield f"{r['ts'].isoformat()},{r['price']},{r['volume']}\n"
    return StreamingResponse(gen(), media_type="text/csv")
```

This keeps memory flat regardless of row count — the analytics-side equivalent of "never hold all items in
memory at once" from R-SCALE §A.

## 10. The repository / service module pattern <a name="10-the-repository-pattern"></a>

The repository is the **only place SQL strings live**. It takes the pool, exposes intention-revealing
methods, returns Pydantic models, and is the unit you mock in tests.

```python
# app/repositories/ohlcv_repo.py
from __future__ import annotations
from datetime import datetime
import asyncpg
from app.schemas.ohlcv import OhlcvBar

class OhlcvRepository:
    """Owns all OHLCV SQL against the TimescaleDB hypertable. The route owns HTTP; this owns SQL."""

    # SQL kept as module/class constants — reviewable, greppable, testable in isolation.
    _OHLCV = """
        SELECT time_bucket($1::interval, ts)          AS bucket,
               first(price, ts)                         AS open,
               max(price)                               AS high,
               min(price)                               AS low,
               last(price, ts)                          AS close,
               sum(volume)                              AS volume
        FROM   ticks
        WHERE  symbol = $2 AND ts >= $3 AND ts < $4
        GROUP  BY bucket
        ORDER  BY bucket
    """

    def __init__(self, pool: asyncpg.Pool) -> None:
        self._pool = pool

    async def fetch_ohlcv(
        self, *, symbol: str, start: datetime, end: datetime, bucket: str
    ) -> list[OhlcvBar]:
        rows = await self._pool.fetch(self._OHLCV, bucket, symbol, start, end)
        return [OhlcvBar.model_validate(dict(r)) for r in rows]

    async def latest_price(self, *, symbol: str) -> Decimal | None:
        # Reads from a continuous aggregate (or the hypertable) — still just SQL.
        return await self._pool.fetchval(
            "SELECT last(price, ts) FROM ticks WHERE symbol = $1", symbol
        )
```

Why this shape:

- **Testable.** Mock `pool.fetch` and assert the repository maps rows correctly, with no DB — the Python
  analogue of Lumina mocking Prisma in `backend-testing`.
- **One source of SQL truth.** A query-plan regression or an index addition is a one-file change.
- **No HTTP leakage.** The repository raises domain errors (`ValueError`, a custom `NotFound`); the route
  translates them to HTTP status codes. SQL never sees a `Request`; HTTP never sees a `Record`.

For a richer service (multi-step: read metadata, fan out reads, assemble), put that orchestration in a
`*_service.py` that *uses* repositories — keeping repositories as thin SQL adapters and the service as the
use-case layer.

## 11. Bulk ingest via COPY <a name="11-bulk-ingest-via-copy"></a>

Loading market data (backfills, nightly EOD, tick streams) must use the **`COPY` protocol**, not
`INSERT … executemany`. COPY is the fastest bulk path; `executemany` is "the slowest by far"
([DEV: speed up bulk inserts with COPY](https://dev.to/josethz00/speed-up-your-postgresql-bulk-inserts-with-copy-40pk)).

### 11.1 asyncpg: `copy_records_to_table`

Signature ([asyncpg API ref](https://magicstack.github.io/asyncpg/current/api/index.html)):
`Connection.copy_records_to_table(table_name, *, records, columns=None, schema_name=None, where=None,
timeout=None)` — `records` is "an iterable returning row tuples to copy into the table. Asynchronous
iterables are also supported."

```python
async def bulk_insert_ticks(pool, ticks: list[tuple]) -> int:
    # ticks: list of (ts, symbol, price, volume) tuples
    async with pool.acquire() as conn:
        result = await conn.copy_records_to_table(
            "ticks",
            records=ticks,
            columns=["ts", "symbol", "price", "volume"],
            timeout=60,
        )
    # result is a status string like "COPY 100000"
    return int(result.split()[-1])
```

The records are **tuples in column order matching `columns`**. asyncpg binds Python types to the COPY
binary stream. This is dramatically faster than row-by-row inserts because there is one round trip and no
per-row plan/parse.

### 11.2 The staging-table upsert pattern (when you need ON CONFLICT)

`COPY` has no `ON CONFLICT`. For idempotent backfills (re-running a day must not double-insert), COPY into
a **staging table**, then `INSERT … SELECT … ON CONFLICT DO NOTHING` into the hypertable — both inside one
transaction ([staging-table upsert pattern](https://overflow.no/blog/2025/1/5/using-staging-tables-for-faster-bulk-upserts-with-python-and-postgresql/)):

```python
async def upsert_ticks(pool, ticks: list[tuple]) -> None:
    async with pool.acquire() as conn:
        async with conn.transaction():
            await conn.execute(
                "CREATE TEMP TABLE _staging (LIKE ticks INCLUDING DEFAULTS) ON COMMIT DROP"
            )
            await conn.copy_records_to_table(
                "_staging", records=ticks, columns=["ts", "symbol", "price", "volume"]
            )
            await conn.execute(
                "INSERT INTO ticks (ts, symbol, price, volume) "
                "SELECT ts, symbol, price, volume FROM _staging "
                "ON CONFLICT (symbol, ts) DO NOTHING"
            )
```

This makes re-ingest **idempotent** — the R-SCALE "heavy ingest, partial-failure behavior" check. A
crashed loader can safely re-run the same batch.

### 11.3 Where ingest lives (the non-negotiable #4 analogue)

Bulk ingest is **heavy, scheduled work** — it belongs in a **dedicated worker/cron process**, not on a
FastAPI request path. A request that triggers a multi-million-row COPY will time out and hold a connection
for minutes. Run ingest as:

- a standalone async script invoked by cron (`asyncio.run(main())`), or
- a worker process consuming a queue/stream,

with its **own** small pool (or even single `connect()`), separate from the API pool. State the runtime
and the partial-failure behavior (staging + `ON CONFLICT` = safe re-run) in the ingest module's docstring.

### 11.4 psycopg3 COPY (the alternative)

psycopg3's COPY is a context manager with `write_row`, and supports binary
([psycopg3 COPY docs](https://www.psycopg.org/psycopg3/docs/basic/copy.html)):

```python
async with conn.cursor() as cur:
    async with cur.copy("COPY ticks (ts, symbol, price, volume) FROM STDIN (FORMAT BINARY)") as copy:
        async for record in tick_source():          # async generator of tuples
            await copy.write_row(record)
```

Binary is "more efficient than textual format, but requires more care with data types, as the server will
not perform an innocent int4 -> int8 cast for you"
([psycopg3 COPY docs](https://www.psycopg.org/psycopg3/docs/basic/copy.html)) — match Python types to
column types exactly when using `FORMAT BINARY`.

## 12. SQLAlchemy 2.0 async — when you do want the ORM <a name="12-sqlalchemy-async"></a>

Use SQLAlchemy-async for the **relational metadata** side (instruments, dashboards, users, saved
queries), and/or if the team strongly prefers an ORM. The hypertable can still be mapped — Timescale's
chunking is transparent to the ORM, which sees a normal table.

### 12.1 Engine + session factory

```python
# app/db/sa.py
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession

engine = create_async_engine(
    "postgresql+asyncpg://user:pass@host/analytics",
    echo=False,
    pool_size=5, max_overflow=10,           # SQLAlchemy's own pool (direct DB)
    pool_pre_ping=True,                      # validate a conn before use (defends stale conns)
)

# expire_on_commit=False avoids lazy-reload of attributes after commit (async footgun)
SessionLocal = async_sessionmaker(engine, expire_on_commit=False, class_=AsyncSession)
```

([SQLAlchemy asyncio docs](https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html) — `async_sessionmaker`,
`expire_on_commit=False`.)

### 12.2 FastAPI session dependency

```python
from typing import AsyncIterator
from sqlalchemy.ext.asyncio import AsyncSession

async def get_session() -> AsyncIterator[AsyncSession]:
    async with SessionLocal() as session:
        yield session            # one session per request; closed on response
```

### 12.3 Mapping a hypertable as a normal table

```python
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column
from sqlalchemy import String, Numeric, TIMESTAMP
from datetime import datetime
from decimal import Decimal

class Base(DeclarativeBase):
    pass

class Tick(Base):
    __tablename__ = "ticks"                 # this IS the hypertable; ORM doesn't know/care
    ts:     Mapped[datetime] = mapped_column(TIMESTAMP(timezone=True), primary_key=True)
    symbol: Mapped[str]      = mapped_column(String, primary_key=True)
    price:  Mapped[Decimal]  = mapped_column(Numeric(18, 8))
    volume: Mapped[Decimal]  = mapped_column(Numeric(20, 8))
```

> The ORM **cannot** express `create_hypertable`, compression, or retention policies — those are DDL the
> ORM/Alembic-autogenerate doesn't know about and must go in a migration (§13). The ORM only ever does
> DML (`SELECT/INSERT/UPDATE/DELETE`) on the table.

### 12.4 Hyperfunctions in the ORM = `text()`

The ORM has no `time_bucket` construct, so analytical queries drop to `text()` — confirming why raw SQL
(§10) is the better home for them:

```python
from sqlalchemy import text

async def hourly_vol(session: AsyncSession, symbol: str, start, end):
    result = await session.execute(
        text(
            "SELECT time_bucket('1 hour', ts) AS bucket, sum(volume) AS v "
            "FROM ticks WHERE symbol = :symbol AND ts >= :start AND ts < :end "
            "GROUP BY bucket ORDER BY bucket"
        ),
        {"symbol": symbol, "start": start, "end": end},   # named binds (:name) for text()
    )
    return result.mappings().all()        # list[RowMapping] → dicts
```

Note `text()` uses **named** binds (`:symbol`), not asyncpg's `$1`. `result.mappings().all()` yields
dict-like rows ready for `Model.model_validate`.

### 12.5 The async ORM caveats checklist

- **`expire_on_commit=False`** — or every attribute access after commit triggers an (illegal-in-async)
  lazy reload.
- **No implicit lazy loading.** Eager-load relationships with `selectinload()` or access via
  `AsyncAttrs.awaitable_attrs` ([SQLAlchemy asyncio docs](https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html)).
  Time-series tables have no relationships, so this only bites the metadata models.
- **One session per request.** Never share an `AsyncSession` across requests or tasks — it is not
  concurrency-safe.
- **Behind a transaction pooler**, set `prepared_statement_cache_size=0` + `connect_args` + `NullPool`
  (§5.4).
- **Across event loops** (rare; e.g. test setup), call `await engine.dispose()` before reuse
  ([SQLAlchemy asyncio docs](https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html)).

## 13. Alembic + hypertables: DDL that the ORM can't autogenerate <a name="13-alembic-hypertables"></a>

Hypertable creation and all Timescale policies are **raw SQL in migrations**, executed with
`op.execute()`. Alembic autogenerate does **not** know about hypertables and will actively fight you (it
tries to drop the index Timescale auto-creates — see below).

### 13.1 The migration: create table, then make it a hypertable

```python
# alembic/versions/0002_ticks_hypertable.py
from alembic import op
import sqlalchemy as sa

revision = "0002_ticks_hypertable"
down_revision = "0001_init"

def upgrade() -> None:
    # 1. ordinary CREATE TABLE — Alembic can autogenerate this part
    op.create_table(
        "ticks",
        sa.Column("ts", sa.TIMESTAMP(timezone=True), nullable=False),     # NOT NULL: required
        sa.Column("symbol", sa.String(), nullable=False),
        sa.Column("price", sa.Numeric(18, 8), nullable=False),
        sa.Column("volume", sa.Numeric(20, 8), nullable=False),
    )

    # 2. convert to a hypertable — RAW SQL, Timescale function (new by_range API, v2.13+)
    op.execute(
        "SELECT create_hypertable("
        "  'ticks', by_range('ts', INTERVAL '1 day'),"
        "  if_not_exists => TRUE"
        ")"
    )

    # 3. composite index for the (symbol, ts) access pattern
    op.create_index("ix_ticks_symbol_ts", "ticks", ["symbol", sa.text("ts DESC")])

    # 4. policies — also raw SQL
    op.execute("SELECT add_compression_policy('ticks', compress_after => INTERVAL '7 days')")
    op.execute("SELECT add_retention_policy('ticks', drop_after => INTERVAL '2 years')")

def downgrade() -> None:
    op.execute("SELECT remove_retention_policy('ticks', if_exists => TRUE)")
    op.execute("SELECT remove_compression_policy('ticks', if_exists => TRUE)")
    op.drop_index("ix_ticks_symbol_ts", table_name="ticks")
    op.drop_table("ticks")     # dropping the table drops the hypertable + chunks
```

API anchors for the SQL above:

- **`create_hypertable('ticks', by_range('ts', INTERVAL '1 day'))`** — the **new generalized API**
  (`by_range`/`by_hash` dimension builders) introduced in **TimescaleDB v2.13**
  ([create_hypertable docs](https://www.tigerdata.com/docs/api/latest/hypertable/create_hypertable);
  [by_range/by_hash note](https://github.com/timescale/docs/blob/latest/api/hypertable/create_hypertable.md)).
  `if_not_exists => TRUE` "print[s] a warning if `relation` is already a hypertable" instead of erroring
  (idempotent re-runs). The time column **must be `NOT NULL`**.
- **Legacy form** (pre-2.13 or by preference):
  `SELECT create_hypertable('ticks', 'ts', chunk_time_interval => INTERVAL '1 day', if_not_exists => TRUE)`
  ([legacy interface](https://docs.timescale.com/api/latest/hypertable/create_hypertable_old/)). The
  positional `time_column_name` ('ts') replaces the `by_range('ts')` builder.
- **`add_compression_policy('ticks', compress_after => INTERVAL '7 days')`** and
  **`add_retention_policy('ticks', drop_after => INTERVAL ...)`** — both plain SQL
  ([add_compression_policy / add_retention_policy](https://www.tigerdata.com/docs/reference/timescaledb/continuous-aggregates/add_policies)).
  Note retention's older positional form `add_retention_policy('ticks', INTERVAL '2 years')` also works.

### 13.2 The autogenerate trap — Alembic tries to drop Timescale's index

When you `create_hypertable`, Timescale **auto-creates an index on the time column**. Alembic's
autogenerate doesn't know that index exists in a migration, so on the *next* autogenerate it emits a
spurious `op.drop_index(...)` for it
([Alembic discussion #1465](https://github.com/sqlalchemy/alembic/discussions/1465)). Two defenses:

**(a)** Pass `create_default_indexes => FALSE` to `create_hypertable` so Timescale makes no surprise
index, and create your own explicitly (then Alembic knows about it):

```python
op.execute(
    "SELECT create_hypertable('ticks', by_range('ts'), "
    "create_default_indexes => FALSE, if_not_exists => TRUE)"
)
op.create_index("ix_ticks_ts", "ticks", [sa.text("ts DESC")])   # now Alembic owns it
```

**(b)** Filter Timescale's internal schemas out of autogenerate so it never reflects chunk tables, in
`env.py`:

```python
# alembic/env.py
def include_object(obj, name, type_, reflected, compare_to):
    # ignore TimescaleDB internal schemas & chunk objects
    if getattr(obj, "schema", None) in {"_timescaledb_internal", "_timescaledb_catalog",
                                        "_timescaledb_config", "_timescaledb_cache"}:
        return False
    return True

context.configure(
    connection=connection,
    target_metadata=target_metadata,
    include_object=include_object,
    include_schemas=False,        # don't reflect the timescale-managed schemas
)
```

([Alembic `include_object`/`include_schemas` mechanism](https://github.com/sqlalchemy/alembic/issues/733).)

### 13.3 Continuous aggregates in a migration

A continuous aggregate is a materialized view created with `WITH (timescaledb.continuous)` — pure SQL via
`op.execute`. It **cannot be created inside a transaction**, so set the migration to non-transactional for
that step, or run the `CREATE MATERIALIZED VIEW` in its own migration with
`op.get_bind().execution_options(isolation_level="AUTOCOMMIT")`:

```python
def upgrade() -> None:
    op.execute(
        """
        CREATE MATERIALIZED VIEW ohlcv_1h
        WITH (timescaledb.continuous) AS
        SELECT time_bucket('1 hour', ts) AS bucket,
               symbol,
               first(price, ts) AS open, max(price) AS high,
               min(price)  AS low,  last(price, ts) AS close,
               sum(volume) AS volume
        FROM ticks
        GROUP BY bucket, symbol
        WITH NO DATA
        """
    )
    op.execute(
        "SELECT add_continuous_aggregate_policy('ohlcv_1h', "
        "start_offset => INTERVAL '3 days', end_offset => INTERVAL '1 hour', "
        "schedule_interval => INTERVAL '1 hour')"
    )
```

`WITH NO DATA` avoids a long backfill at migration time; refresh it afterward in a controlled job
(`CALL refresh_continuous_aggregate('ohlcv_1h', NULL, now())`).

### 13.4 Migration engine driver

Run Alembic with a **sync** driver to keep the migration env simple
(`postgresql+psycopg://...` or `postgresql+psycopg2://...`), even if the app runs async. Alembic supports
async too, but a sync migration engine sidesteps event-loop wiring in `env.py` and runs DDL fine. Keep the
**DSN out of source** — read it from env in `env.py` (`config.set_main_option("sqlalchemy.url",
os.environ["DATABASE_URL"])`).

## 14. psycopg3 — the sync+async alternative <a name="14-psycopg3"></a>

Choose psycopg3 when you want **one driver for sync (cron/Alembic) and async (API)** and richer COPY/SQL
composition. It is slower than asyncpg on the hot path but very capable.

### 14.1 Async connection + query

```python
import psycopg
from psycopg.rows import dict_row

async with await psycopg.AsyncConnection.connect(DSN, row_factory=dict_row) as conn:
    async with conn.cursor() as cur:
        await cur.execute(
            "SELECT time_bucket('1 hour', ts) AS bucket, sum(volume) AS v "
            "FROM trades WHERE symbol = %s AND ts >= %s AND ts < %s "
            "GROUP BY bucket ORDER BY bucket",
            (symbol, start, end),               # psycopg uses %s placeholders + a tuple
        )
        rows = await cur.fetchall()             # list[dict] thanks to dict_row
```

psycopg uses `%s` positional placeholders (not asyncpg's `$1`). `row_factory=dict_row` gives dict rows
directly — convenient for `Model.model_validate`.

### 14.2 The psycopg3 async pool

`psycopg_pool.AsyncConnectionPool` — open it explicitly in lifespan with `open=False`, because *"opening
an async pool in the constructor is deprecated"* and *"you should call `await pool.open()` or `async with
... as pool`"* ([psycopg3 pool docs](https://www.psycopg.org/psycopg3/docs/advanced/pool.html)):

```python
from psycopg_pool import AsyncConnectionPool

@asynccontextmanager
async def lifespan(app: FastAPI):
    pool = AsyncConnectionPool(conninfo=DSN, min_size=2, max_size=10, open=False)
    await pool.open()                 # explicit open (not in constructor)
    app.state.pool = pool
    yield
    await pool.close()

# in a route/repo:
async with app.state.pool.connection() as conn:     # checkout
    async with conn.cursor(row_factory=dict_row) as cur:
        await cur.execute("SELECT ...", params)
        rows = await cur.fetchall()
```

Pool params: `min_size` (default 4), `max_size`, `max_lifetime`, `max_idle`, `timeout`
([psycopg3 pool docs](https://www.psycopg.org/psycopg3/docs/advanced/pool.html)).

### 14.3 psycopg3 COPY (see §11.4)

Covered above — `async with cur.copy("COPY ... FROM STDIN") as copy: await copy.write_row(rec)`.

### 14.4 Safe dynamic identifiers with `psycopg.sql`

When you genuinely need a dynamic table/column name, psycopg3's `sql` module composes injection-safe SQL:

```python
from psycopg import sql

query = sql.SQL("SELECT {col} FROM {tbl} WHERE symbol = %s").format(
    col=sql.Identifier(column_name),     # safely quoted identifier
    tbl=sql.Identifier(table_name),
)
await cur.execute(query, (symbol,))      # value still a bound param
```

`sql.Identifier` quotes/escapes the identifier — the typed alternative to the allow-list of §8.2 (use the
allow-list anyway for *which* identifiers are permitted; use `sql.Identifier` for *rendering* them).

## 15. Health checks, timeouts, retries, observability <a name="15-health-checks"></a>

### 15.1 Liveness/readiness

```python
@router.get("/healthz")
async def healthz(pool: asyncpg.Pool = Depends(get_pool)) -> dict:
    try:
        await pool.fetchval("SELECT 1")           # cheap, no prepared-statement issue
        return {"db": "ok"}
    except Exception:
        raise HTTPException(status_code=503, detail="db unavailable")
```

### 15.2 Timeouts at three levels

1. **Pool/connection** — `command_timeout=30.0` on `create_pool` (per-operation ceiling).
2. **Per-query** — `await pool.fetch(sql, *args, timeout=5.0)` for an endpoint with a tight SLA.
3. **Postgres-side** — `SET LOCAL statement_timeout = '5s'` inside a transaction, so the **server** kills
   a runaway query even if the client gives up (prevents orphaned heavy scans on the DB).

A time-range analytics endpoint over a large hypertable should set a server-side `statement_timeout` so a
pathological range can't pin a backend.

### 15.3 Retries — only for transient errors

Retry **connection** errors (`asyncpg.exceptions.ConnectionDoesNotExistError`,
`InterfaceError`) and serialization failures with backoff; **never** blindly retry a query that may have
side effects unless it is idempotent. For the read-only analytics path, a bounded retry (2–3 attempts,
exponential backoff) on connection-level errors is safe. Reads are idempotent; writes must be idempotent
by construction (staging + `ON CONFLICT`, §11.2) before they're retry-safe.

### 15.4 Observability

- Set `application_name` per service (§3.3) so `pg_stat_activity` attributes connections.
- Log slow queries (`pg_stat_statements` on the DB; `log_min_duration_statement`).
- Track pool saturation: `pool.get_size()`, `pool.get_idle_size()` (asyncpg) — alert when idle hits 0
  (you're connection-starved → raise `max_size` or add read replicas, the R-SCALE "scale reads without
  touching writes" lever).

## 16. Anti-patterns quick table <a name="16-anti-patterns"></a>

| Anti-pattern | Why it breaks | Fix |
|---|---|---|
| SQL string inside a FastAPI route | Layer violation; untestable; spreads SQL everywhere | Move to a repository method (§1, §10) |
| New pool/connection per request | Connection churn, exhausts `max_connections`, latency | One pool in lifespan on `app.state` (§4) |
| f-string a value into SQL | SQL injection + no plan reuse | Bound params `$1`/`%s` (§8) |
| Bind a table/column name as a parameter | Can't bind identifiers; silently fails or you fall back to f-string | Allow-list enum / `sql.Identifier` (§8.2, §14.4) |
| asyncpg prepared statements ON behind a transaction pooler | `prepared statement … does not exist` under load | `statement_cache_size=0` (+ SQLAlchemy `prepared_statement_cache_size=0`) (§5) |
| `INSERT … executemany` for bulk loads | Slowest path; saturates the loop | `copy_records_to_table` / `cursor.copy()` (§11) |
| COPY then expecting `ON CONFLICT` | COPY has no upsert | Staging table + `INSERT … SELECT … ON CONFLICT` (§11.2) |
| Cast `numeric` price to `float` on the way out | Precision loss = invented money number | Keep `Decimal` end-to-end (§9.3) |
| `create_hypertable` / policies via ORM autogenerate | ORM can't model it; autogen drops Timescale's index | `op.execute("SELECT create_hypertable(...)")` + index filter (§13) |
| Unbounded `start..end` over a 1M-row hypertable | Self-DoS; full chunk scan | Cap span in Pydantic + index `(symbol, ts)` + chunk-prune on time (§8.3–8.4) |
| ORM session shared across requests/tasks | Not concurrency-safe; corrupt state | One `AsyncSession` per request (§12.5) |
| Heavy ingest on a request path | Times out, holds a connection minutes | Run ingest in a cron/worker with its own pool (§11.3) |
| `expire_on_commit=True` (default) in async | Post-commit attribute access triggers illegal lazy reload | `expire_on_commit=False` (§12.1) |
| `refresh_continuous_aggregate` inside `conn.transaction()` | It's a procedure; can't run in a txn | `pool.execute("CALL refresh_continuous_aggregate(...)")` outside a txn (§7.3) |
